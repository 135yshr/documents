### 🔴 重大な問題

**1. 注文保存とイベント発行の「二重書き込み（Dual Write）」問題**

376〜387行目：

```go
if err := uc.orderRepo.Save(ctx, order); err != nil { ... }
// ↓ ここで障害が起きたら？
if err := uc.publisher.Publish(ctx, e); err != nil { ... }
```

`Save` が成功した後に `Publish` が失敗すると、注文は永続化されたのにイベントが発行されず、在庫が減りません。これは分散システムにおける典型的な「Dual Write」問題です。TOCTOU注記は書かれていますが、この問題はTOCTOUとは別の障害パターンです。

Outboxパターン（注文と同じトランザクションでイベントをDBに書き込み、別プロセスで配信する）や、少なくとも「インメモリのEventBusではこの問題は顕在化しにくいが、分散環境では追加の考慮が必要」という注記が必要です。

---

**2. `inventory` が `order` の内部パッケージを直接インポートしている**

466行目：

```go
import "myapp/internal/order/event"
```

511行目のチェックリストで「コンテキスト間で `internal` パッケージを直接インポートしない」と書かれていますが、このコードがまさにそれを行っています。

410行目の説明では「ネストした `internal` の外に置く」としてこの構成を正当化していますが、コード例はフラットな `internal` 構成であり、ネストした `internal` は使っていません。フラット構成で `order/event/` はたしかにGoのコンパイラ上はアクセス可能ですが、チェックリストの「直接インポートの禁止」ルールとの矛盾が残ります。チェックリスト側を「**コンテキストの `internal` パッケージ（domain/usecase）を直接インポートしない。公開API（event/port）のみ許可する**」のように修正するか、区別を明確にすべきです。

---

### 🟠 設計上の問題

**3. `DomainEvent.ID` に `order.ID` を代入している**

380行目：

```go
DomainEvent: event.DomainEvent{ID: order.ID, OccurredAt: time.Now()},
OrderID:     order.ID,
```

`DomainEvent.ID` はイベント自体の一意識別子であるべきですが、ここでは注文IDを入れています。`OrderID` フィールドが別に存在するため、2つのフィールドに同じ値が入り、イベントIDと注文IDが混同されます。イベントIDは `uuid.New().String()` などで独立して生成すべきです。

---

**4. `HandleOrderPlaced` が在庫不足をチェックせずに減算している**

479行目：

```go
return h.stockRepo.Decrease(ctx, placed.ProductID, placed.Quantity)
```

TOCTOU注記で「別の注文が在庫を消費する可能性」に触れていますが、ハンドラ側で在庫不足時の振る舞い（エラー返却、補償トランザクション、アラート発報等）が一切考慮されていません。`Decrease` が在庫0以下になる場合にどうなるかが不明で、負の在庫を許容するのかどうかの設計判断が示されていません。

---

**5. イベント購読の文字列ベースルーティングが脆弱**

491行目：

```go
eventBus.Subscribe(ctx, "OrderPlaced", handler.Handle)
```

マジックストリング `"OrderPlaced"` で購読していますが、`Publish` 側ではどのようにイベント型を文字列に変換するのかが示されていません。EventBusの実装がリフレクションで型名を取得するのか、イベント側が `EventType() string` メソッドを持つのかが不明です。この文字列がずれるとイベントが静かに消失するため、型ベースのルーティングか定数定義への言及が必要です。

---

### 🟡 軽微な指摘

**6. パターン1とパターン2が混在した `PlaceOrderUseCase`**

現在の `PlaceOrderUseCase` は `StockChecker`（パターン1：同期連携）と `EventPublisher`（パターン2：非同期連携）の両方を持っています。記事の構成では「パターン1：共有インターフェースによる連携」セクション内にこのコードがありますが、実際にはパターン2のイベント発行も含んでいます。読者はパターン1と2を比較検討したいのに、コード例が両方を混ぜているため、それぞれのトレードオフが見えにくくなっています。

---
