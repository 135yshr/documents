# ドキュメントレビュー

引数で指定された Markdown 記事ファイルの文章品質をレビューし、問題を指摘したうえで自動修正する。

## 入力

この７つのドキュメントを対象にレビューしてください に対象ファイルのパスが渡される（例: `articles/978121945958ed.md`）。パスが渡されない場合は「対象ファイルを指定してください（例: /review-doc articles/xxx.md）」とだけ返して終了すること。

## レビュー手順

### Step 1: 記事を読み込む

対象ファイルを Read ツールで全文読み込むこと。

### Step 2: 以下の 4 観点でレビューする

#### 観点 A: 冗長・繰り返し

- 同じ主張が複数箇所で繰り返されていないか
- 削れる前置き・つなぎ表現はないか（「〜がある。それは〜である。」のような回りくどい表現）
- 1 文が長すぎないか（目安: 80 文字超は分割を検討）

#### 観点 B: 論理構成・流れ

- セクション間の論理的なつながりが自然か
- 見出し階層が適切か（h2 → h3 の飛び越しがないか）
- 結論が導入の問いに対応しているか

#### 観点 C: 読者価値密度

- 各セクションに具体例・数値・コード例など「読んで良かった」と思える情報があるか
- 抽象的な主張だけで終わっているセクションはないか
- 読者が次のアクションを取れる情報が含まれているか

#### 観点 D: 文体の統一

- 記事全体が敬語（ですます調）で統一されているか
- 箇条書き内で「である」調が混在していないか
- 見出し直下の地の文とリスト項目で文体が揺れていないか

### Step 3: 指摘を出力する

以下のフォーマットで指摘を一覧表示すること。

```markdown
## レビュー結果: <ファイル名>

### 観点 A: 冗長・繰り返し

- [ ] L<行番号>: <指摘内容>（<改善案の要約>）

### 観点 B: 論理構成・流れ

- [ ] L<行番号>: <指摘内容>（<改善案の要約>）

### 観点 C: 読者価値密度

- [ ] L<行番号>: <指摘内容>（<改善案の要約>）

### 観点 D: 文体の統一

- [ ] L<行番号>: <指摘内容>（<改善案の要約>）

### 総評

<全体の品質に関する 2〜3 行のコメント>
```

指摘がない観点は「問題なし」とだけ記載すること。

### Step 4: 自動修正を適用する

指摘のうち、意味を変えずに改善できるもの（冗長表現の削除、文の分割など）は Edit ツールで自動修正すること。修正を適用したら、修正箇所のサマリーを出力すること。

内容の追加や構成の大幅な変更が必要な指摘は、修正案を提示するにとどめ、自動修正しないこと。

### Step 5: lint を実行する

修正後に以下を順に実行し、エラーがあれば修正すること。

```bash
npx prettier --write <対象ファイル>
npx markdownlint-cli2 <対象ファイル>
npx textlint <対象ファイル>
```

## 制約

- フロントマター（title, emoji, topics, published など）は変更しないこと
- 技術用語・固有名詞はそのまま残すこと
- 修正は最小限にとどめ、筆者の文体を尊重すること
- published の値は変更しないこと

# Smart Commit with Gitmoji

- If currently on `main` or `master` branch, create and checkout a new feature branch with a descriptive name

1. **Review all changes** using `git status` and `git diff --staged` (or `git diff` if nothing staged)
2. **Stage changes** if not already staged using `git add .` or selective staging
3. **Analyze the changes** to understand:
   - Type of changes (new feature, bug fix, refactor, etc.)
   - Scope of changes (which files/components affected)
   - Impact and purpose of the changes

<optional detailed explanation if needed>

- **Body**: Explain "what" and "why", not "how"
- **Language**: Write in English
- **Tone**: Professional and concise

## レビュー：批判的分析

### 🔴 重大な問題

**1. `Cancel()` の状態遷移と Mermaid 図の矛盾**

Mermaid 図では `Confirmed → Cancel → Cancelled` の遷移が示されていますが、`Cancel()` のコードは「すでにキャンセル済みなら拒否」するだけで、**確定済み注文のキャンセル可否を明示的に扱っていません**。

```go
func (o *Order) Cancel() error {
    if o.status == OrderStatusCancelled {
        return errors.New("すでにキャンセル済みです")
    }
    o.status = OrderStatusCancelled  // Confirmed からもキャンセルできてしまう
    return nil
}
```

EC サイトでは「確定済み注文はキャンセル不可」というビジネスルールが一般的です。図と実装のどちらが意図した仕様なのか、記事内で明示するか、少なくともコメントで補足すべきです。

---

**2. `items` スライスの防御的コピー不足**

`NewOrder` は渡された `items` スライスの参照をそのまま保持します。

```go
// 呼び出し元がスライスを変更できてしまう
items := []model.OrderItem{item1}
order, _ := model.NewOrder("id-1", items)
items[0] = somethingElse  // order の内部状態が書き換わる
```

コンストラクタで `copy()` するか、新しいスライスを生成すべきです。「不変条件を守る」という記事のテーマと直接矛盾します。

---

### 🟡 設計上のトレードオフが未整理

**3. `Snapshot()` とドメイン層の純粋性**

記事は `Snapshot` を「ゲッター最小化と整合する」と紹介していますが、**ドメインモデルが永続化・APIレスポンスの関心事を知ることになる**という設計上の問題を無視しています。

`OrderSnapshot` の構造が永続化スキーマやAPIの形に引きずられると、ドメインモデルがインフラ・プレゼンテーション層に依存する逆転が起きます。これはクリーンアーキテクチャの文脈で特に重要な論点であり、メリットだけでなくこのトレードオフを議論すべきです。

---

**4. `Status()` ゲッターを "しない" としながら `Snapshot()` で同等情報を外部に出す矛盾**

「`status` のゲッターは公開しない」としているのに、`Snapshot()` では `Status` フィールドとして同じ値を外部に渡しています。「ゲッターを避ける理由 = 外部での条件分岐を防ぐ」が目的なら、`Snapshot` 経由でも同じリスクは残ります。この点を論理的に説明するか、整合性を取る必要があります。

---

**5. `ReconstructOrder` の `internal` 活用が「絵に描いた餅」**

`internal` パッケージの活用を提案していますが、具体例がありません。実際には `domain/model` パッケージと `infrastructure/postgres` パッケージの関係上、`internal` パッケージをどう配置するかは非自明です。

```
// internal パッケージで本当に制限できる構成例がないと
// 読者が実装しようとして詰まる
```

「検討します」で終わらせず、実際の構成例を示すか、難易度に正直に言及すべきです。

---

### 🟢 軽微な指摘

**6. `totalPrice` に `int` を使うリスク**

通貨計算で `int` を使うと、32bit 環境では約21億円が上限になります。`int64` を明示する方が安全で意図が明確です。これは「設計判断」として一言触れる価値があります。

**7. 並行安全性への言及がない**

ポインタレシーバを使ったメソッド（`AddItem`、`Confirm` 等）は goroutine から同時に呼ばれると競合状態になります。DDD 文脈で集約は「トランザクション境界」でもあるため、楽観的ロックや mutex が不要な理由（or 必要な場合の言及）があると記事の信頼性が上がります。

**8. `NewOrderItem` で `name` の空文字チェックがない**

`productID` は検証しているのに `name` は検証していません。意図的な設計なら理由を、見落としなら修正を。

**9. まとめの「Java/C# と同等のカプセル化」は言い過ぎ**

直前の比較表でパッケージレベルとクラスレベルの違いを説明しておきながら、「同等」と結論付けるのは論理的に矛盾します。「同等の効果を実現できる」や「実用上は十分な保護が得られる」程度の表現にとどめるべきです。

## 優先修正案

| 優先度 | 箇所 | 修正内容 |
|--------|------|----------|
| 🔴 高 | `Cancel()` メソッド | Mermaid 図との整合性確認・仕様の明示 |
| 🔴 高 | `NewOrder` | `items` スライスの防御的コピー追加 |
| 🟡 中 | `Snapshot()` セクション | ドメイン層汚染のトレードオフを追記 |
| 🟡 中 | `Snapshot` vs ゲッターの矛盾 | `status` の扱いを論理的に整合 |

全体的な構成と説明の質は高く、標準ライブラリを使った根拠付けも効果的です。上記の特に `Cancel()` の状態遷移とスライスの防御的コピーは、コードの正確性に関わるため優先して確認してください。

# Smart Commit with Gitmoji

Execute the following steps non-interactively:

## Branch Management

- If currently on `main` or `master` branch, create and checkout a new feature branch with a descriptive name
- If on any other branch, proceed with commit on current branch (no branch creation)
- If no changes are detected, exit without doing anything

## Change Analysis & Commit

1. **Review all changes** using `git status` and `git diff --staged` (or `git diff` if nothing staged)
2. **Stage changes** if not already staged using `git add .` or selective staging
3. **Analyze the changes** to understand:
   - Type of changes (new feature, bug fix, refactor, etc.)
   - Scope of changes (which files/components affected)
   - Impact and purpose of the changes

## Commit Message Creation

Create a commit message following this format:

```
<gitmoji> <type>: <concise description>

<optional detailed explanation if needed>
```

### Gitmoji Selection Guide

Choose the most appropriate gitmoji from <https://gitmoji.dev/> based on change type:

**Common Patterns:**

- ✨ `:sparkles:` - New features
- 🐛 `:bug:` - Bug fixes
- 🎨 `:art:` - Code structure/format improvements
- ⚡️ `:zap:` - Performance improvements
- 📝 `:memo:` - Documentation updates
- 🔧 `:wrench:` - Configuration changes
- ♻️ `:recycle:` - Code refactoring
- 🚨 `:rotating_light:` - Fix linter warnings
- ✅ `:white_check_mark:` - Add/update tests
- 🔒️ `:lock:` - Security fixes
- ⬆️ `:arrow_up:` - Upgrade dependencies
- 🚚 `:truck:` - Move/rename files
- 🗑️ `:wastebasket:` - Remove deprecated code

### Message Guidelines

- **Title**: Keep under 50 characters, use imperative mood
- **Body**: Explain "what" and "why", not "how"
- **Language**: Write in English
- **Tone**: Professional and concise

## Final Steps

- Execute the commit with the generated message
- Display a summary of what was committed
- Suggest next steps (push to remote, create PR, etc.)
