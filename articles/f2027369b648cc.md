## はじめに

:::message

本記事は私の実装経験と考察をベースにしています。各セクションの根拠となる一次情報源は、該当箇所に参照リンクを記載しています。

:::

Goでクリーンアーキテクチャを導入すると、ほぼ確実にこう感じる瞬間が来ます。

> 「interface多すぎない？」

UserRepository, UserUseCase, UserService, UserPresenter... レイヤーごとにinterfaceと実装のペアが増殖し、気がつくと「1つの機能を追加するのに5ファイル以上触る」という状態になります。

この記事では、Goでクリーンアーキテクチャを実践する中で**interfaceが爆発する原因**を分析し、**実務で使える処方箋**を提示します。

---

## なぜinterfaceが爆発するのか

### 原因1：教科書をそのまま適用している

クリーンアーキテクチャの解説記事では、依存性逆転の原則（DIP）を示すために**すべての境界にinterfaceを置く**例が多く紹介されます。

```text
❌ 教科書的に全境界にinterfaceを置いた結果

internal/
├── domain/user/
│   ├── user.go
│   ├── repository.go          # interface
│   └── service.go             # interface
├── application/user/
│   ├── usecase.go             # interface
│   ├── usecase_impl.go        # 実装
│   ├── input_port.go          # interface
│   └── output_port.go         # interface
├── infrastructure/user/
│   ├── mysql_repository.go    # 実装
│   └── user_service_impl.go   # 実装
└── presentation/user/
    ├── handler.go             # 実装
    └── presenter.go           # 実装
```

1つのUserエンティティに対して、interface 5個、実装 4個。合計9ファイルです。

### 原因2：「将来のために」抽象化している

```go
// 実装が1つしかないのにinterfaceを定義
type UserPresenter interface {
    Output(user *User) *UserResponse
}

// この実装しか存在しない
type JSONUserPresenter struct{}

func (p *JSONUserPresenter) Output(user *User) *UserResponse {
    return &UserResponse{Name: user.Name()}
}
```

「いつかXML出力が必要になる」と考えてinterfaceを作りますが、その「いつか」は大抵来ません。

### 原因3：Goのimplicit interfaceの利点を活かせていない

JavaやC#では、interfaceを事前に定義しないと依存性逆転ができません。一方、Goではinterfaceの満足は暗黙的（implicit）です。**利用側がinterfaceを定義する**のがGoの慣習です。

> Accept interfaces, return structs.
>
> — Go Proverb（Jack Lindamood, [Preemptive Interface Anti-Pattern in Go](https://medium.com/@cep21/preemptive-interface-anti-pattern-in-go-54c18ac0668a)）

## 処方箋1：interfaceは「利用側」で定義する

Goの標準ライブラリがこの原則の最良の手本です。`io.Reader`はデータを読む**利用側**のパッケージで使われるinterfaceであり、データを提供する側（`os.File`や`bytes.Buffer`）はinterfaceの存在を知りません。

### Before：提供側でinterfaceを定義

```go
// domain/user/repository.go
package user

// 提供側が大きなinterfaceを定義
type Repository interface {
    FindByID(ctx context.Context, id UserID) (*User, error)
    FindByEmail(ctx context.Context, email Email) (*User, error)
    FindAll(ctx context.Context) ([]*User, error)
    Save(ctx context.Context, user *User) error
    Delete(ctx context.Context, id UserID) error
}
```

すべてのUseCaseがこの大きなinterfaceに依存し、実際にはFindByIDしか使わないUseCaseでも5メソッド全体への依存を強いられます。

### After：利用側で必要なメソッドだけ定義

```go
// application/get_user.go
package application

// 利用側が必要な分だけ定義
type userFinder interface {
    FindByID(ctx context.Context, id user.UserID) (*user.User, error)
}

type GetUserUseCase struct {
    finder userFinder
}

func (uc *GetUserUseCase) Execute(ctx context.Context, id user.UserID) (*user.User, error) {
    return uc.finder.FindByID(ctx, id)
}
```

```go
// application/delete_user.go
package application

type userDeleter interface {
    Delete(ctx context.Context, id user.UserID) error
}

type DeleteUserUseCase struct {
    deleter userDeleter
}
```

infrastructure層の`MySQLUserRepository`は両方のinterfaceを**暗黙的に**満たします。interfaceの定義ファイルを別途作る必要はありません。

## 処方箋2：Interface Segregation Principle（ISP）を適用する

処方箋1の背景にある原則が、SOLID原則の**Interface Segregation Principle（インターフェース分離の原則）**です。

> Clients should not be forced to depend upon interfaces that they do not use.
>
> — Robert C. Martin, _Agile Software Development_

Goではimplicit interfaceの仕組みにより、この原則を自然に実践できます。

```go
// ❌ 太いinterface（Fat Interface）
type UserRepository interface {
    FindByID(ctx context.Context, id UserID) (*User, error)
    FindByEmail(ctx context.Context, email Email) (*User, error)
    FindAll(ctx context.Context) ([]*User, error)
    Save(ctx context.Context, user *User) error
    Delete(ctx context.Context, id UserID) error
}

// ✅ 細いinterface（Thin Interface）
type UserReader interface {
    FindByID(ctx context.Context, id UserID) (*User, error)
}

type UserWriter interface {
    Save(ctx context.Context, user *User) error
}

// 必要なら合成もできる
type UserReadWriter interface {
    UserReader
    UserWriter
}
```

### 判断基準：interfaceのメソッド数

| メソッド数 | 判断                                                  |
| ---------- | ----------------------------------------------------- |
| 1〜2       | 理想的。Go標準ライブラリのinterfaceの多くがこのサイズ |
| 3〜4       | 許容範囲。ただし全メソッドを使う利用者がいるか確認    |
| 5以上      | 分割を検討。太いinterfaceの兆候                       |

## 処方箋3：UseCase層のinterfaceは本当に必要か問う

```go
// ❌ UseCaseにinterfaceを定義しがち
type GetUserUseCase interface {
    Execute(ctx context.Context, id UserID) (*User, error)
}

type GetUserUseCaseImpl struct {
    finder userFinder
}
```

`GetUserUseCaseImpl`をモックしたい場面はどこでしょうか。Handler（Presentation層）のテストでしょうか。

多くのプロジェクトでは、Handler層のテストはHTTPリクエストを投げる統合テストで行います。その場合、UseCaseをモックする必要はありません。

```go
// ✅ interfaceなしのUseCase
type GetUserUseCase struct {
    finder userFinder
}

func (uc *GetUserUseCase) Execute(ctx context.Context, id user.UserID) (*user.User, error) {
    return uc.finder.FindByID(ctx, id)
}
```

### UseCaseにinterfaceが必要なケース

- Handler層のユニットテストで**UseCaseをモックしたい**場合
- 複数のUseCaseが**共通のインターフェース**として扱われる場合（コマンドパターン等）

これらに該当しないなら、structだけで十分です。

## 処方箋4：Presenter / OutputPortを削る

クリーンアーキテクチャの原典では、Presenter / OutputPortの概念が登場します。しかしGoのWebアプリケーションでは、ほとんどの場合**不要**です。

### Before：Presenterを律儀に実装

```go
// application/output_port.go
type GetUserOutputPort interface {
    Output(user *User) error
}

// presentation/presenter.go
type GetUserPresenter struct {
    w http.ResponseWriter
}

func (p *GetUserPresenter) Output(user *User) error {
    return json.NewEncoder(p.w).Encode(UserResponse{Name: user.Name()})
}
```

### After：UseCaseがDTOを返し、Handlerが変換する

```go
// application/get_user.go
func (uc *GetUserUseCase) Execute(ctx context.Context, id user.UserID) (*UserDTO, error) {
    u, err := uc.finder.FindByID(ctx, id)
    if err != nil {
        return nil, err
    }
    return &UserDTO{ID: u.ID().String(), Name: u.Name()}, nil
}

// presentation/handler.go
func (h *UserHandler) GetUser(w http.ResponseWriter, r *http.Request) {
    dto, err := h.getUser.Execute(r.Context(), userID)
    if err != nil {
        http.Error(w, err.Error(), http.StatusInternalServerError)
        return
    }
    json.NewEncoder(w).Encode(dto)
}
```

Presenterを挟まなくても、**レイヤー間の依存方向は正しく保たれます**。UseCaseはHTTPの存在を知りません。

## 処方箋5：ディレクトリ構成を整理する

処方箋1〜4を適用した結果のディレクトリ構成を示します。

### Before：9ファイル

```text
internal/
├── domain/user/
│   ├── user.go
│   ├── repository.go          # interface
│   └── service.go             # interface
├── application/user/
│   ├── usecase.go             # interface
│   ├── usecase_impl.go
│   ├── input_port.go          # interface
│   └── output_port.go         # interface
├── infrastructure/user/
│   ├── mysql_repository.go
│   └── user_service_impl.go
└── presentation/user/
    ├── handler.go
    └── presenter.go           # 実装
```

### After：5ファイル

```text
internal/
├── domain/user/
│   └── user.go                # 集約（interfaceなし）
├── application/user/
│   ├── get_user.go            # UseCase（利用側でinterfaceを定義）
│   ├── create_user.go
│   └── dto.go
├── infrastructure/user/
│   └── mysql_repository.go    # 実装（domain層のinterfaceを知らない）
└── presentation/user/
    └── handler.go             # Presenterは不要
```

**interface定義用の独立ファイルがゼロ**になりました。interfaceは各UseCaseファイル内にprivateで定義されています。

## よくある反論への回答

### 「domain層にRepository interfaceを置かないと依存性逆転にならないのか」

GoのISPに従い、利用側（application層）でinterfaceを定義すれば、依存性は逆転します。

```text
application/get_user.go
    ↓ 定義
    userFinder interface
    ↑ 暗黙的に実装
infrastructure/mysql_repository.go
```

domain層へinterfaceを置くパターン（DDD原典のRepository）は、**ドメインロジックがRepositoryへ直接依存するとき**に有効です。Go × クリーンアーキテクチャでは、ドメインロジックではなくUseCaseがRepositoryを呼ぶケースが多いため、application層での定義が自然です。

ただし、**ドメインサービス内でRepositoryを使う場合**は、domain層にinterfaceを置く従来の方法が適切です。

```go
// domain/user/service.go
// ドメインサービスがRepositoryに依存する場合は、domain層にinterfaceを置く
type Repository interface {
    FindByID(ctx context.Context, id UserID) (*User, error)
}

type DuplicateChecker struct {
    repo Repository
}

func (c *DuplicateChecker) IsDuplicate(ctx context.Context, email Email) (bool, error) {
    // ドメインロジックの中でRepositoryを使う
    existing, err := c.repo.FindByEmail(ctx, email)
    if err != nil {
        return false, err
    }
    return existing != nil, nil
}
```

### 「テストでモックできなくなるのか」

利用側でinterfaceを定義するので、**テストに必要なモックは必ず作れます**。むしろ、テスト対象が実際に使うメソッドだけのモックになるため、モックが簡潔になります。

```go
// application/get_user_test.go
type mockFinder struct {
    user *user.User
    err  error
}

func (m *mockFinder) FindByID(_ context.Context, _ user.UserID) (*user.User, error) {
    return m.user, m.err
}

func TestGetUser(t *testing.T) {
    uc := &GetUserUseCase{
        finder: &mockFinder{user: testUser, err: nil},
    }
    result, err := uc.Execute(context.Background(), testUserID)
    // ...
}
```

### 「チームで統一できなくなるのか」

ADR（Architecture Decision Record）や`ARCHITECTURE.md`にルールを明記します。

```markdown
## interface定義のルール

1. interfaceは利用側で定義する（提供側には定義しない）
2. 1つのinterfaceは1〜2メソッドを目安にする
3. domain層にinterfaceを置くのは、ドメインサービスが依存するときのみ
4. UseCase層にinterfaceを定義しない（structで十分）
```

## 判断フローチャート

```text
interfaceを作るべき？

1. そのinterfaceを使う側（利用者）は誰？
   → 利用者が明確でない → ❌ 作らない

2. 利用者は複数の実装を切り替える必要がある？
   → Yes → ✅ 利用側で定義する
   → No → 3へ

3. テストでモックが必要？
   → Yes → ✅ テストファイルと同じパッケージで定義する
   → No → ❌ 作らない（structで十分）
```

## まとめ

| 処方箋 | 内容 | 削減効果 |
| --- | --- | --- |
| 1. 利用側で定義 | interfaceは使う側が定義する | interface定義ファイルの削減 |
| 2. ISPを適用 | 1〜2メソッドの細いinterfaceにする | 不要な依存の削減 |
| 3. UseCase interfaceを削る | structで十分。モックが必要な場合のみ定義 | ファイル数の削減 |
| 4. Presenterを削る | DTOを返してHandlerで変換する | レイヤー数の削減 |
| 5. ディレクトリ整理 | 処方箋1〜4の結果を構成に反映 | 9ファイル → 5ファイル |

Goでクリーンアーキテクチャを導入する際は、**教科書をそのまま移植するのではなく、Goの言語特性を活かした設計**を心がけましょう。interfaceの爆発は、「Goの流儀に合わない抽象化」のサインです。
