## はじめに

Claude Code を日常的に使っていると、トークン消費量が気になってきます。設定ファイルの肥大化、不要なプラグイン、冗長なスキル定義など、知らないうちにトークンを浪費しているケースは多いです。

この記事では、実際に Claude Code の作業環境を体系的に見直した手順を紹介します。各改善について実測データを示しながら、読者が自分の環境で同じ最適化を実践できることを目指しています。

## 対象読者

- Claude Code を日常的に使っている開発者
- トークン消費量を減らしたい方
- Claude Code の設定を体系的に見直したい方

## 最適化の全体像

実施した改善は大きく6つのカテゴリに分かれます。

| カテゴリ | 改善内容 | 効果 | エビデンス |
| --- | --- | --- | --- |
| CLAUDE.md のスリム化 | 8,924バイト → 4,440バイト | 推定約1,300トークン削減（後述） | バイト数実測・トークン数は推定 |
| プラグインの整理 | 不要な5つを無効化 | スキル注入頻度の低減（効果は未計測） | 未計測 |
| ユーザースキルの整理 | 重複・不要な3つを削除 | description 読み込み削減（効果は未計測） | 未計測 |
| スキルの圧縮 | commit スキル 66行 → 17行 | ファイルサイズ削減（効果は未計測） | 行数のみ実測 |
| グローバル設定の整理 | 不要データ削除 | 微量 | - |
| メモリの活用 | 行動指針を記録 | 作業効率向上（定性的） | 定量評価不可 |

> **注意**: 本記事の数値はすべて観測ベースの推定であり、Claude Code の内部仕様に依存するため再現性は保証されません。トークン削減量が推定できているのは CLAUDE.md のスリム化のみです。その他の項目は理論上の効果であり、実際のトークン削減量は環境や使い方によって異なります。

## 1. CLAUDE.md を3層ドキュメント設計でスリム化する

### 問題

`CLAUDE.md` は多くの場合、セッション開始時に参照対象として読み込まれます（内部挙動は非公開）。ここが肥大化すると、**トークン消費の増加につながる可能性**があります。

よくある問題は以下のとおりです。

- コードから読み取れる情報（フォント設定、テーマ、ESLint設定）を書いている
- PR テンプレートの全文を埋め込んでいる（`.github/PULL_REQUEST_TEMPLATE.md` に同じ内容があるのに）
- 詳細なアーキテクチャ説明を書いている（`docs/architecture.md` があるのに）

### 3層ドキュメント設計とは

```text
第1層: CLAUDE.md（セッション開始時に参照される）
  → 最小限の情報 + 第2層へのリンク

第2層: docs/（必要時のみ参照）
  → 詳細な仕様書、設計書、ガイド

第3層: .claude/, memory/（運用・設定情報）
  → スキル定義、メモリ（MEMORY.md も参照対象）、エージェント設定
```

**ポイント**: 第1層には「Claude が毎回知るべき最低限の情報」だけを書き、詳細は第2層にリンクで参照させます。

### 実践: CLAUDE.md のスリム化

**Before（261行、8,924バイト）:**

```markdown
## Architecture & Code Structure

### Project Structure

src/ app/ # App Router directory **tests**/ # Component tests layout.tsx # Root layout with Geist fonts ...（ディレクトリツリー全文）

### Font Configuration

The app uses Geist fonts loaded via next/font/google:

- --font-geist-sans for primary text
- --font-geist-mono for monospace text

### Theme System

Uses CSS custom properties for theming:

- --background and --foreground with automatic dark mode detection ...
```

**After（118行、4,440バイト）:**

```markdown
## Architecture

- Path alias: `@/*` maps to `./src/*`
- Tests: `__tests__/` directories, `*.test.ts(x)` naming, Jest + React Testing Library

詳細は以下を参照:

- [アーキテクチャ設計](docs/architecture.md)
- [API 仕様概要](docs/api/00-overview.md)
- [ダッシュボードチャート実装ガイド](docs/dashboard-charts-guide.md)
```

**実測結果:**

| 指標         | Before       | After        | 削減率 |
| ------------ | ------------ | ------------ | ------ |
| バイト数     | 8,924        | 4,440        | 50%    |
| 行数         | 261          | 118          | 55%    |
| 推定トークン | 2,500〜3,600 | 1,300〜1,800 | 約50%  |

> **計測方法**: `wc -c CLAUDE.md` でバイト数を計測。トークン数は筆者の環境（日本語混在テキスト）で `/context` コマンドの表示値とバイト数を比較し、おおよそ1トークン=2.5〜3.5バイトの範囲で推定しました。ただし、トークン数は言語や記号構成によって大きく変動するため、この値はあくまで特定環境での目安です。トークナイザーの仕様は非公開のため、正確な値は `/context` コマンドで確認してください。

### 削除の判断基準

| 情報の種類                        | 判断     | 理由                                          |
| --------------------------------- | -------- | --------------------------------------------- |
| コマンド一覧（dev, test, lint）   | 残す     | 毎回参照する                                  |
| パスエイリアス設定                | 残す     | import に影響                                 |
| フォント設定の詳細                | 削除     | `layout.tsx` を読めばわかる                   |
| テーマのCSS変数名                 | 削除     | `globals.css` を読めばわかる                  |
| ESLint 設定の詳細                 | 削除     | `.eslintrc` を読めばわかる                    |
| PR テンプレート全文               | リンク化 | `.github/PULL_REQUEST_TEMPLATE.md` に同じ内容 |
| 重要なルール（`bun test` 禁止等） | 残す     | 毎回守るべきルール                            |

## 2. プラグインをプロジェクト別に管理する

### 全プロジェクトでプラグインが有効になる問題

`~/.claude/settings.json` で有効化したプラグインは**全プロジェクトで利用可能な状態**になります。たとえば Go プロジェクト用の `gopls-lsp` が Next.js プロジェクトでも有効だと、不要なスキル・ツールのロードにつながります。

> **注意**: プラグイン無効化によるトークン削減量は未計測です。プラグインのスキルは「必要なときだけ自動注入」される仕組みのため、常にトークンを消費するわけではありません。ただし、無効化することで不要なスキルが注入されるリスクを排除できます。

### プロジェクト別に無効化する

プロジェクトルートの `.claude/settings.json` に `enabledPlugins` を追加します。

```json
{
  "enabledPlugins": {
    "gopls-lsp@claude-plugins-official": false,
    "deploy-on-aws@claude-plugins-official": false,
    "terraform@claude-plugins-official": false,
    "claude-code-setup@claude-plugins-official": false,
    "slack@claude-plugins-official": false
  }
}
```

### 設定の優先順位

```text
プロジェクト .claude/settings.json  ← 最優先
  ↓
グローバル ~/.claude/settings.json  ← フォールバック
```

グローバルで `true` にしておき、不要なプロジェクトでのみ `false` にするのがおすすめです。

### プラグイン整理の判断基準

プロジェクトの技術スタックと照合して判断します。

```text
このプロジェクト: Next.js + TypeScript + Tailwind CSS

必要:
  ✅ typescript-lsp（TypeScript開発）
  ✅ context7（ドキュメント参照）
  ✅ chrome-devtools-mcp（ブラウザデバッグ）

不要:
  ❌ gopls-lsp（Go言語用）
  ❌ deploy-on-aws（AWS用、Vercel使用）
  ❌ terraform（IaC用）
```

## 3. ユーザースキルを整理する

### スキルの重複と肥大化の問題

`~/.claude/commands/` にあるユーザースキルは、セッション開始時に description が参照される可能性があります。プラグインが提供するスキルと重複していたり、使わなくなったスキルが残っていたりすると、不要なトークン消費につながります。

> **注意**: 筆者が確認した範囲では、参照されるのは各スキルの description（frontmatter の1行）のみです。スキル本文は `/スキル名` で呼び出したときに読み込まれます。

### 重複チェックの方法

```bash
# ユーザースキルの一覧
ls ~/.claude/commands/

# プラグインスキルとの比較
# /context コマンドで User / Plugin のスキルを確認
```

実例として、`frontend-design` スキルの重複がありました。`~/.claude/commands/frontend-design.md` とプラグイン版が完全に同じ内容だったため、ユーザー側を削除しました。

### スキルの圧縮

使い続けるスキルも、冗長な記述を圧縮できます。

**Before（66行）:**

```markdown
### Gitmoji Selection Guide

Choose the most appropriate gitmoji from https://gitmoji.dev/:

**Common Patterns:**

- ✨ `:sparkles:` - New features
- 🐛 `:bug:` - Bug fixes
- 🎨 `:art:` - Code structure/format improvements ...（各項目に説明文付き）

### Message Guidelines

- **Title**: Keep under 50 characters, use imperative mood
- **Body**: Explain "what" and "why", not "how"
- **Language**: Write in English
- **Tone**: Professional and concise
```

**After（17行、一部抜粋）:**

```markdown
...（手順1〜4: ブランチ確認、diff確認、ステージング、メッセージ作成）

Gitmoji: ✨ feat / 🐛 fix / ♻️ refactor / 🎨 format / ⚡️ perf / 📝 docs / 🔧 config / ✅ test / 🚨 lint / 🔒 security / ⬆️ deps / 🚚 move / 🗑️ remove

5. Commit and show summary with suggested next steps.
```

gitmoji の意味は一般的な知識であり、詳細な説明は不要です。**プロジェクト固有でない一般知識は書かない**のが圧縮のコツです。

> **注意**: スキル本文の圧縮は、呼び出し時のトークン消費を減らします。ただし、description が同じなら一覧表示時の消費は変わりません。

## 4. グローバル設定をクリーンに保つ

### 不要データの混入チェック

`~/.claude/settings.json` に内部状態データが混入していることがあります。

```json
{
  "feedbackSurveyState": {
    "lastShownTime": 1754101429950
  }
}
```

`feedbackSurveyState` はサーベイの表示履歴で、設定として不要です。このようなデータは削除しましょう。

### フックで毎回の指示を自動化する

毎回同じ指示を書くのが面倒な場合、`UserPromptSubmit` フックで `additionalContext` を返すことで自動追加できます。

```json
{
  "hooks": {
    "UserPromptSubmit": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "printf '{\"additionalContext\": \"指示内容をここに記述\"}'"
          }
        ]
      }
    ]
  }
}
```

> **補足**: 筆者の環境（macOS, zsh）では `echo` も `printf` も Claude Code の JSON パーサーで正常に解析できました。唯一の違いは `echo` が末尾に改行を付加する点です。ただし、`echo` はシェルや環境によって挙動が異なる場合があるため（`-e` オプションのデフォルト有効化など）、JSON 出力には挙動が一貫している `printf` を使う方が安全です。

## 5. メモリを活用して行動パターンを改善する

### メモリの種類と用途

Claude Code のメモリシステム（`~/.claude/projects/*/memory/`）に保存した情報は、セッション開始時に参照される可能性があります。

メモリファイルの frontmatter に `type` フィールドを指定できます。筆者の運用では、以下の分類を使用するとメモリの整理や参照がしやすくなりました。

| 種類        | 用途                   | 例                                  |
| ----------- | ---------------------- | ----------------------------------- |
| `feedback`  | Claude の行動指針      | 「破壊的操作は必ず確認を取る」      |
| `project`   | プロジェクト固有の情報 | 「Phase 5 は未マージ」              |
| `reference` | 外部リソースへの参照   | 「バックエンドは port 9000 で起動」 |
| `user`      | ユーザーの背景情報     | 「Go歴10年、React初心者」           |

### 実例: 曖昧なプロンプトへの対応

自分のプロンプトの癖（短すぎる指示、対象が不明瞭）を分析し、Claude が自動的に確認を取るようメモリに記録しました。

```markdown
---
name: 曖昧なプロンプトへの対応
description: ユーザーのプロンプトが曖昧・短い場合は対象と意図を確認してから作業する
type: feedback
---

ユーザーのプロンプトが曖昧な場合、推測で作業を進めず対象と意図を確認する。

**Why:** 「削除してください」「はい」「1」など短いプロンプトで文脈依存の指示が多く、意図しない処理や不要な作業が発生した。

**How to apply:**

- 番号のみの回答 → 「X番の〇〇ですね」と対象を復唱して確認
- 「削除/変更してください」に対象がない → 何を削除/変更するか確認
- ただし直前の選択肢への回答など文脈が明確な場合は確認不要
```

> **トレードオフ**: メモリ追加で推定50〜100トークン程度の増加が見込まれます。ただし、意図しない作業のやり直しを防ぐことで、結果的にセッション全体のトークン消費を抑える効果が期待できます（定量的な検証はできていません）。

### プロンプトの癖を分析する方法

セッション履歴の JSONL ファイルから自分のプロンプトを抽出できます。

```bash
# 最新セッションのプロンプトを抽出
latest=$(ls -t ~/.claude/projects/*/YOUR_PROJECT/*.jsonl | head -1)
python3 -c "
import json
with open('$latest') as f:
    for line in f:
        obj = json.loads(line.strip())
        if obj.get('type') == 'user' and not obj.get('isMeta', False):
            content = obj.get('message', {}).get('content', '')
            if isinstance(content, str) and not content.startswith('<'):
                print(f'USER: {content[:200]}')
                print('---')
"
```

> **注意**: このスクリプトは Claude Code の JSONL 内部形式（`type`, `isMeta`, `message` 等のフィールド）に依存しています。この形式は公式ドキュメントで仕様として公開されているものではなく、筆者が 2026年4月時点のファイル構造を解析して作成したものです。Claude Code のバージョンアップで形式が変更され、動作しなくなる可能性が高い点にご注意ください。

これで自分のプロンプトを一覧できます。短すぎる指示、対象が不明瞭な指示を探して改善しましょう。

## 6. 定期的な見直しをスキル化する

ここまでの最適化を毎回手動でやるのは面倒です。定期的なレビューをスキル化しました。

```markdown
---
description: Claude Code の作業環境を定期的に見直し、トークン効率と設定の最適化を行う
---

# 作業環境の最適化レビュー

## 1. CLAUDE.md のスリム化チェック

- 行数とトークン量を確認
- コードから読み取れる情報が含まれていないか
- docs/ へのリンクで参照できているか

## 2. プラグイン見直し

- プロジェクトの技術スタックに不要なプラグインがないか

## 3. ユーザースキル見直し

- プラグインスキルと重複するスキルがないか
- 圧縮可能なスキルがないか

## 4. グローバル設定の確認

- 不要な内部状態データが混入していないか

## 5. メモリファイルの整理

- 古くなった情報や解決済みの issue がないか
```
