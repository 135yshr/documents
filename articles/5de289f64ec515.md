## はじめに

:::message

本記事は私の実装経験と考察をベースにしています。各セクションの根拠となる一次情報源は、該当箇所に参照リンクを記載しています。

:::

GoでクリーンアーキテクチャやDDDを導入したプロジェクトで、こんなコードを見たことはないでしょうか。

```go
// application/get_user.go
type GetUserUseCase struct {
    repo UserRepository
}

func (uc *GetUserUseCase) Execute(ctx context.Context, id string) (*User, error) {
    return uc.repo.FindByID(ctx, id)
}
```

UseCaseの中身が**Repositoryの呼び出し1行だけ**。いわゆる「パススルーUseCase」です。

このようなUseCaseがプロジェクト全体の半分以上を占めている場合、UseCase層は本当に必要なのでしょうか。この記事では、UseCase層がパススルーになる原因を分析し、**GoでのクリーンアーキテクチャにおけるUseCase層の要否の判断基準**を提示します。

---

## UseCase層がパススルーになる原因

### 原因1：CRUDが中心のアプリケーション

多くの業務アプリケーションは、機能の大半がCRUD（作成・読み取り・更新・削除）です。CRUDにはビジネスロジックがほとんどなく、「受け取ったデータをそのまま保存する」「IDで取得して返す」という処理になります。

```go
// 全部パススルー
func (uc *CreateUserUseCase) Execute(ctx context.Context, input CreateUserInput) error {
    user := user.New(input.Name, input.Email)
    return uc.repo.Save(ctx, user)
}

func (uc *GetUserUseCase) Execute(ctx context.Context, id string) (*UserDTO, error) {
    u, err := uc.repo.FindByID(ctx, id)
    if err != nil {
        return nil, err
    }
    return toDTO(u), nil
}

func (uc *UpdateUserUseCase) Execute(ctx context.Context, id string, input UpdateUserInput) error {
    u, err := uc.repo.FindByID(ctx, id)
    if err != nil {
        return err
    }
    u.UpdateName(input.Name)
    return uc.repo.Save(ctx, u)
}

func (uc *DeleteUserUseCase) Execute(ctx context.Context, id string) error {
    return uc.repo.Delete(ctx, id)
}
```

4つのUseCaseを作りましたが、どれもRepositoryを呼ぶだけです。

### 原因2：「いつか複雑になる」という先回り

「今はCRUDだけど、将来ビジネスロジックが入る」と考えてUseCase層を用意するケースです。しかし、その「将来」が来なければ、パススルーのまま残り続けます。

> You Ain't Gonna Need It（YAGNI）
>
> — Ron Jeffries, _Extreme Programming Installed_

### 原因3：教科書の構成をそのまま適用している

クリーンアーキテクチャの解説では4層（Presentation / Application / Domain / Infrastructure）が標準的に紹介されます。この構成を**すべての機能に一律適用**すると、CRUD操作にもUseCase層が強制されます。

## UseCase層が必要になるケース

UseCase層が**真の価値を発揮するケース**を整理します。

### ケース1：複数のドメイン操作を組み合わせる

```go
// ✅ UseCase層に価値がある：注文の確定処理
type PlaceOrderUseCase struct {
    orderRepo    order.Repository
    stockService stock.Service
    eventBus     event.Bus
}

func (uc *PlaceOrderUseCase) Execute(ctx context.Context, input PlaceOrderInput) error {
    // 1. 在庫を確認する
    if err := uc.stockService.Reserve(ctx, input.Items); err != nil {
        return err
    }

    // 2. 注文を作成する
    o, err := order.Place(input.CustomerID, input.Items)
    if err != nil {
        return err
    }

    // 3. 注文を保存する
    if err := uc.orderRepo.Save(ctx, o); err != nil {
        return err
    }

    // 4. イベントを発行する
    uc.eventBus.Publish(ctx, order.PlacedEvent{OrderID: o.ID()})

    return nil
}
```

複数のドメイン操作をオーケストレーションする場所として、UseCase層は不可欠です。

### ケース2：トランザクション境界の管理

```go
// ✅ UseCase層に価値がある：トランザクション制御
type TransferMoneyUseCase struct {
    accountRepo account.Repository
    txManager   transaction.Manager
}

func (uc *TransferMoneyUseCase) Execute(ctx context.Context, from, to account.ID, amount int) error {
    return uc.txManager.RunInTx(ctx, func(ctx context.Context) error {
        sender, err := uc.accountRepo.FindByID(ctx, from)
        if err != nil {
            return err
        }
        receiver, err := uc.accountRepo.FindByID(ctx, to)
        if err != nil {
            return err
        }

        if err := sender.Withdraw(amount); err != nil {
            return err // ドメインエラー：残高不足
        }
        receiver.Deposit(amount)

        if err := uc.accountRepo.Save(ctx, sender); err != nil {
            return err
        }
        return uc.accountRepo.Save(ctx, receiver)
    })
}
```

### ケース3：認可・権限チェック

```go
// ✅ UseCase層に価値がある：認可チェック
type DeleteProjectUseCase struct {
    projectRepo project.Repository
    authz       authorization.Service
}

func (uc *DeleteProjectUseCase) Execute(ctx context.Context, projectID string) error {
    actor := auth.ActorFrom(ctx)
    if err := uc.authz.Can(ctx, actor, "project:delete", projectID); err != nil {
        return err
    }
    return uc.projectRepo.Delete(ctx, project.ID(projectID))
}
```

### ケース4：入力の変換・バリデーション

```go
// ✅ UseCase層に価値がある：入力からドメインオブジェクトへの変換
type CreateUserUseCase struct {
    repo        userRepository
    hashService password.HashService
}

func (uc *CreateUserUseCase) Execute(ctx context.Context, input CreateUserInput) (*UserDTO, error) {
    email, err := user.NewEmail(input.Email)
    if err != nil {
        return nil, err
    }
    hashedPassword, err := uc.hashService.Hash(input.Password)
    if err != nil {
        return nil, err
    }
    u := user.New(input.Name, email, hashedPassword)
    if err := uc.repo.Save(ctx, u); err != nil {
        return nil, err
    }
    return toDTO(u), nil
}
```

## 判断基準：UseCase層を置くか、省略するか

```text
その機能にビジネスロジックはある？

1. 複数のドメイン操作を組み合わせる？
   → Yes → ✅ UseCase層が必要

2. トランザクション制御が必要？
   → Yes → ✅ UseCase層が必要

3. 認可チェックが必要？
   → Yes → ✅ UseCase層が必要

4. 入力→ドメインオブジェクトの変換ロジックがある？
   → Yes → ✅ UseCase層が必要

5. すべてNoで、Repositoryの単純な呼び出しだけ？
   → ❌ UseCase層は省略可能
```

## UseCase層を省略した構成

CRUD中心の機能ではUseCase層を省略し、Handler層からRepositoryを直接使うパターンがあります。

### パターン：HandlerがRepositoryを直接使う

```go
// presentation/user/handler.go
type UserHandler struct {
    repo userRepository
}

// 利用側でinterfaceを定義
type userRepository interface {
    FindByID(ctx context.Context, id string) (*user.User, error)
    Save(ctx context.Context, u *user.User) error
    Delete(ctx context.Context, id string) error
}

func (h *UserHandler) GetUser(w http.ResponseWriter, r *http.Request) {
    id := r.PathValue("id")
    u, err := h.repo.FindByID(r.Context(), id)
    if err != nil {
        HandleError(w, err)
        return
    }
    if u == nil {
        writeJSON(w, http.StatusNotFound, ErrorResponse{Message: "ユーザーが見つかりません"})
        return
    }
    writeJSON(w, http.StatusOK, toResponse(u))
}

func (h *UserHandler) DeleteUser(w http.ResponseWriter, r *http.Request) {
    id := r.PathValue("id")
    if err := h.repo.Delete(r.Context(), id); err != nil {
        HandleError(w, err)
        return
    }
    w.WriteHeader(http.StatusNoContent)
}
```

:::message alert

この方法は**CRUDのみの機能**に限定してください。ビジネスロジックがHandler層に漏れ出すと、テストが困難になり、ロジックの重複も発生します。

:::

### 依存の方向は正しいか

HandlerがRepositoryに直接依存しても、interfaceを利用側（Handler）で定義しているため、**依存性逆転の原則は守られています**。

```text
presentation/user/handler.go
    ↓ 定義
    userRepository interface
    ↑ 暗黙的に実装
infrastructure/user/mysql_repository.go
```

domain層がhandler層のinterfaceに依存することはないため、依存の方向は内側→外側の一方通行が維持されます。

## 混在する構成のディレクトリ例

現実のプロジェクトでは、CRUD機能と複雑な機能が混在します。**機能ごとにUseCase層の有無を判断する**のが実用的です。

```text
internal/
├── domain/
│   ├── user/
│   │   └── user.go
│   └── order/
│       ├── order.go
│       └── order_item.go
│
├── application/
│   └── order/                     # 複雑な機能 → UseCase層あり
│       ├── place_order.go         # 在庫確認 + 注文作成 + イベント発行
│       ├── cancel_order.go        # ステータス検証 + キャンセル + 返金
│       └── dto.go
│
├── infrastructure/
│   ├── user/
│   │   └── mysql_repository.go
│   └── order/
│       └── mysql_repository.go
│
└── presentation/
    ├── user/
    │   └── handler.go             # CRUD → Repositoryを直接使用
    └── order/
        └── handler.go             # 複雑 → UseCaseを使用
```

User機能はCRUDなのでUseCase層なし、Order機能は複雑なビジネスロジックがあるのでUseCase層ありという構成です。

## よくある反論への回答

### 「UseCase層がないと、後からロジックを追加するときに大変か」

Go言語ではリファクタリングのコストは低いです。`gopls`（Go公式のLanguage Server）によるリネームや、コンパイラの型チェックにより、安全に構造を変更できます。

**パススルーUseCase → ロジック付きUseCaseへの変更手順：**

1. `application/user/create_user.go`にUseCaseを作成する
2. Handler内のRepository呼び出しをUseCase呼び出しに変更する
3. コンパイルが通ることを確認する

この変更は短時間で完了します。「将来のため」に空のUseCase層を維持するコスト（ファイル数増加、新規メンバーの学習コスト）と比較すると、必要になってから追加する方が合理的です。

### 「CRUDとビジネスロジックの境界があいまいか」

明確な判断基準があります。

```text
以下のいずれかに該当するか？
  □ Repositoryの呼び出しが2回以上ある
  □ 条件分岐（if/switch）にビジネスルールが含まれる
  □ 他のサービスやドメインサービスを呼ぶ
  □ トランザクション制御が必要
  □ イベントの発行がある

→ 1つでも該当すればUseCase層を作る
→ すべてNoならHandler層で直接処理
```

### 「チーム内で統一感がなくなるのか」

ADRにルールを明記します。

```markdown
## UseCase層の方針

- CRUD操作（Repository呼び出し1回のみ）：Handler層で直接処理
- ビジネスロジックを含む操作：UseCase層を作成
- 判断に迷った場合：UseCase層を作成する（安全側に倒す）
```

「判断に迷ったらUseCase層を作る」というルールで、過度な省略を防ぎます。

## 段階的なアプローチ

いきなり「UseCase層を省略する」と決めるのではなく、**段階的に判断する**のが安全です。

```text
Step 1: まずすべての機能にUseCase層を作る（教科書通り）
        → プロジェクト初期の構成をシンプルに保つ

Step 2: パススルーUseCaseを特定する
        → UseCaseの中身がRepository呼び出し1行だけのものをリスト化

Step 3: パススルーUseCaseを削除し、Handlerから直接呼ぶように変更
        → 1機能ずつ段階的に移行

Step 4: ADRにルールを記録
        → 新規機能の開発時に判断基準を適用
```

## 最小構成のテンプレート

クリーンアーキテクチャの最小構成で始めるためのディレクトリテンプレートです。

```text
internal/
├── domain/                    # ビジネスルール
│   └── {aggregate}/
│       └── {aggregate}.go     # エンティティ + 値オブジェクト
│
├── application/               # 複雑なユースケースのみ
│   └── {aggregate}/
│       ├── {usecase}.go       # 必要な場合のみ作成
│       └── dto.go
│
├── infrastructure/            # 技術的な実装
│   └── {aggregate}/
│       └── {db}_repository.go
│
└── presentation/              # HTTP ハンドラ
    └── {aggregate}/
        └── handler.go
```

**ルール：**

- `domain/` は必ず作る（ビジネスルールの置き場所）
- `application/` は必要になってから作る
- `infrastructure/` は必ず作る（DBアクセス等）
- `presentation/` は必ず作る（HTTPハンドラ等）

## まとめ

| 状況                                  | UseCase層 | 理由                             |
| ------------------------------------- | --------- | -------------------------------- |
| Repository呼び出し1回だけ             | 省略可能  | パススルーは価値を生まない       |
| 複数のドメイン操作を組み合わせる      | 必要      | オーケストレーションの場所が必要 |
| トランザクション制御がある            | 必要      | トランザクション境界の管理       |
| 認可チェックがある                    | 必要      | 横断的関心事の統合ポイント       |
| 入力→ドメインオブジェクトの変換が複雑 | 必要      | 変換ロジックの置き場所           |

UseCase層は「すべての機能に必要なレイヤー」ではなく、**「ビジネスロジックが存在する機能に必要なレイヤー」**です。CRUDにはCRUDに適した構成を、複雑なロジックにはそれに適した構成を。**一律に同じ構造を強制しないこと**が、Goでのクリーンアーキテクチャを実用的にする鍵です。
