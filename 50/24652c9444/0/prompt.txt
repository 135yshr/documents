# ドキュメントレビュー

引数で指定された Markdown 記事ファイルの文章品質をレビューし、問題を指摘したうえで自動修正する。

## 入力

 に対象ファイルのパスが渡される（例: `articles/978121945958ed.md`）。パスが渡されない場合は「対象ファイルを指定してください（例: /review-doc articles/xxx.md）」とだけ返して終了すること。

## レビュー手順

### Step 1: 記事を読み込む

対象ファイルを Read ツールで全文読み込むこと。

### Step 2: 以下の 4 観点でレビューする

#### 観点 A: 冗長・繰り返し

- 同じ主張が複数箇所で繰り返されていないか
- 削れる前置き・つなぎ表現はないか（「〜がある。それは〜である。」のような回りくどい表現）
- 1 文が長すぎないか（目安: 80 文字超は分割を検討）

#### 観点 B: 論理構成・流れ

- セクション間の論理的なつながりが自然か
- 見出し階層が適切か（h2 → h3 の飛び越しがないか）
- 結論が導入の問いに対応しているか

#### 観点 C: 読者価値密度

- 各セクションに具体例・数値・コード例など「読んで良かった」と思える情報があるか
- 抽象的な主張だけで終わっているセクションはないか
- 読者が次のアクションを取れる情報が含まれているか

#### 観点 D: 文体の統一

- 記事全体が敬語（ですます調）で統一されているか
- 箇条書き内で「である」調が混在していないか
- 見出し直下の地の文とリスト項目で文体が揺れていないか

### Step 3: 指摘を出力する

以下のフォーマットで指摘を一覧表示すること。

```markdown
## レビュー結果: <ファイル名>

### 観点 A: 冗長・繰り返し

- [ ] L<行番号>: <指摘内容>（<改善案の要約>）

### 観点 B: 論理構成・流れ

- [ ] L<行番号>: <指摘内容>（<改善案の要約>）

### 観点 C: 読者価値密度

- [ ] L<行番号>: <指摘内容>（<改善案の要約>）

### 観点 D: 文体の統一

- [ ] L<行番号>: <指摘内容>（<改善案の要約>）

### 総評

<全体の品質に関する 2〜3 行のコメント>
```

指摘がない観点は「問題なし」とだけ記載すること。

### Step 4: 自動修正を適用する

指摘のうち、意味を変えずに改善できるもの（冗長表現の削除、文の分割など）は Edit ツールで自動修正すること。修正を適用したら、修正箇所のサマリーを出力すること。

内容の追加や構成の大幅な変更が必要な指摘は、修正案を提示するにとどめ、自動修正しないこと。

### Step 5: lint を実行する

修正後に以下を順に実行し、エラーがあれば修正すること。

```bash
npx prettier --write <対象ファイル>
npx markdownlint-cli2 <対象ファイル>
npx textlint <対象ファイル>
```

## 制約

- フロントマター（title, emoji, topics, published など）は変更しないこと
- 技術用語・固有名詞はそのまま残すこと
- 修正は最小限にとどめ、筆者の文体を尊重すること
- published の値は変更しないこと

# Smart Commit with Gitmoji

Execute the following steps non-interactively:

## Branch Management

- If currently on `main` or `master` branch, create and checkout a new feature branch with a descriptive name
- If on any other branch, proceed with commit on current branch (no branch creation)
- If no changes are detected, exit without doing anything

## Change Analysis & Commit

1. **Review all changes** using `git status` and `git diff --staged` (or `git diff` if nothing staged)
2. **Stage changes** if not already staged using `git add .` or selective staging
3. **Analyze the changes** to understand:
   - Type of changes (new feature, bug fix, refactor, etc.)
   - Scope of changes (which files/components affected)
   - Impact and purpose of the changes

## Commit Message Creation

Create a commit message following this format:

```
<gitmoji> <type>: <concise description>

<optional detailed explanation if needed>
```

### Gitmoji Selection Guide

Choose the most appropriate gitmoji from <https://gitmoji.dev/> based on change type:

**Common Patterns:**

- ✨ `:sparkles:` - New features
- 🐛 `:bug:` - Bug fixes
- 🎨 `:art:` - Code structure/format improvements
- ⚡️ `:zap:` - Performance improvements
- 📝 `:memo:` - Documentation updates
- 🔧 `:wrench:` - Configuration changes
- ♻️ `:recycle:` - Code refactoring
- 🚨 `:rotating_light:` - Fix linter warnings
- ✅ `:white_check_mark:` - Add/update tests
- 🔒️ `:lock:` - Security fixes
- ⬆️ `:arrow_up:` - Upgrade dependencies
- 🚚 `:truck:` - Move/rename files
- 🗑️ `:wastebasket:` - Remove deprecated code

### Message Guidelines

- **Title**: Keep under 50 characters, use imperative mood
- **Body**: Explain "what" and "why", not "how"
- **Language**: Write in English
- **Tone**: Professional and concise

## Final Steps

- Execute the commit with the generated message
- Display a summary of what was committed
- Suggest next steps (push to remote, create PR, etc.)

## 改善提案

### 1. RepositoryをReader/Writerに分離する（前記事の処方箋3と直接対応）

前の記事では「Repositoryが太い」ことを問題として処方箋を提示していましたが、この記事の `domain/order/repository.go` は分離されていません。

**現在のコード:**
```go
type Repository interface {
    FindByID(ctx context.Context, id OrderID) (*Order, error)
    Save(ctx context.Context, order *Order) error
}
```

**前記事の原則を適用した改善案:**
```go
// domain/order/repository.go
type OrderReader interface {
    FindByID(ctx context.Context, id OrderID) (*Order, error)
}

type OrderWriter interface {
    Save(ctx context.Context, order *Order) error
}
```

するとUseCaseの依存が最小化されます:
- `PlaceOrderUseCase` → `OrderWriter` のみに依存（Findは不要）
- `CancelOrderUseCase` → `OrderReader` + `OrderWriter` に依存

これは「The bigger the interface, the weaker the abstraction」の実践例として、この記事のコードサンプル内で示せる機会です。

---

### 2. Presentation層でUseCaseのinterfaceを定義する（前記事の処方箋1と対応）

前の記事で最も効果が大きいとしていた「利用側でinterfaceを定義する」が、この記事のHandlerには適用されていません。

**改善案（Presentation層のhandler.go）:**
```go
// presentation/order/handler.go

// Handlerが必要なメソッドだけinterfaceとして定義する（前記事の原則）
type orderListFetcher interface {
    Execute(ctx context.Context) ([]apporder.OrderListDTO, error)
}
type orderPlacer interface {
    Execute(ctx context.Context, input apporder.PlaceOrderInput) error
}

type OrderHandler struct {
    lister orderListFetcher
    placer orderPlacer
}
```

UseCase側はinterfaceを定義せずstructを公開するだけで済み、前の記事と設計思想が一貫します。前記事との対応関係が明示されていると、2記事を読んでいる読者にとって「なるほど、Handler層でも同じ原則が使える」と腑に落ちやすくなります。

---

### 3. 「QueryServiceのinterfaceを利用側で定義する」という選択肢の補足

現在の記事では `application/order/query_service.go` に `OrderQueryService` インターフェースを置いていますが、前の記事の「処方箋2（Output Port精査）」の考え方と連動させると、こう説明できます：

| 条件 | 方針 |
|---|---|
| 1つのUseCaseしか使わない | そのUseCase内でprivate interfaceとして定義 |
| 複数のUseCaseが共有する | `query_service.go` として共有interfaceにする |

`FindAll` と `FindByID` が別々のUseCaseから呼ばれる場合は現在の共有interfaceが適切ですが、この判断基準を記事内に明示すると、前の記事の処方箋2と整合性が取れます。

# Smart Commit with Gitmoji

Execute the following steps non-interactively:

## Branch Management

- If currently on `main` or `master` branch, create and checkout a new feature branch with a descriptive name
- If on any other branch, proceed with commit on current branch (no branch creation)
- If no changes are detected, exit without doing anything

## Change Analysis & Commit

1. **Review all changes** using `git status` and `git diff --staged` (or `git diff` if nothing staged)
2. **Stage changes** if not already staged using `git add .` or selective staging
3. **Analyze the changes** to understand:
   - Type of changes (new feature, bug fix, refactor, etc.)
   - Scope of changes (which files/components affected)
   - Impact and purpose of the changes

## Commit Message Creation

Create a commit message following this format:

```
<gitmoji> <type>: <concise description>

<optional detailed explanation if needed>
```

### Gitmoji Selection Guide

Choose the most appropriate gitmoji from <https://gitmoji.dev/> based on change type:

**Common Patterns:**

- ✨ `:sparkles:` - New features
- 🐛 `:bug:` - Bug fixes
- 🎨 `:art:` - Code structure/format improvements
- ⚡️ `:zap:` - Performance improvements
- 📝 `:memo:` - Documentation updates
- 🔧 `:wrench:` - Configuration changes
- ♻️ `:recycle:` - Code refactoring
- 🚨 `:rotating_light:` - Fix linter warnings
- ✅ `:white_check_mark:` - Add/update tests
- 🔒️ `:lock:` - Security fixes
- ⬆️ `:arrow_up:` - Upgrade dependencies
- 🚚 `:truck:` - Move/rename files
- 🗑️ `:wastebasket:` - Remove deprecated code

### Message Guidelines

- **Title**: Keep under 50 characters, use imperative mood
- **Body**: Explain "what" and "why", not "how"
- **Language**: Write in English
- **Tone**: Professional and concise

## Final Steps

- Execute the commit with the generated message
- Display a summary of what was committed
- Suggest next steps (push to remote, create PR, etc.)

## 批判的な視点からの指摘候補

### 1. サバイバーシップバイアス（最も強い指摘）

記事の中心的な主張「パススルーのまま残ったInteractorはほとんどなかった」は、筆者の特定プロジェクト（分析・バッチ処理が中心）の経験のみに基づいています。

批判者はこう言えます：「それはあなたのプロジェクトがたまたま複雑化したからであって、シンプルなCRUD中心のアプリなら一生パススルーのままかもしれない。複雑化した事例しか語られておらず、シンプルのまま終わった反例が示されていない」

---

### 2. YAGNIへの反論が論理的に弱い

記事はYAGNIに対して「アーキテクチャの一貫性のためにパススルーは許容できる」と答えていますが、YAGNIの本質的な反論になっていません。

YAGNIの立場からはこう言えます：「一貫性のためなら、複雑化したときに追加すれば良い。Goのimplicit interfaceとリファクタリングツールがあれば、後からInteractor層を差し込むコストは低い。今あえてパススルー層を作るのは、まさしくYAGNI違反そのものだ」

---

### 3. 「依存の方向が崩れる」という主張への反論余地

記事では「HandlerからRepositoryを直接呼ぶとDomain層に直接依存する」と書いていますが、これは設計の工夫で解消できます。

```
// HandlerがRepositoryインターフェースに依存する設計（依存方向は保てる）
type jobFinder interface {
    FindByID(ctx context.Context, id string) (*model.Job, error)
}
type JobHandler struct {
    repo jobFinder  // ← Handler側でinterfaceを定義すれば依存逆転は成立する
}
```

批判者から「UseCase層がなくても依存の方向は崩れない。それはあなたの設計問題であってUseCase層の必要性の根拠にはならない」と言われる余地があります。

---

### 4. 参考文献にVertical Slice Architectureを入れているのに、主張と矛盾している

記事の参考文献にJimmy BogardのVertical Slice Architectureが載っています。しかしVSAは「機能ごとにスライスして考え、水平な層を強制しない」という思想であり、「全機能に統一してUseCase層を持つ」という本記事の主張とは真逆の方向性です。

なぜVSAを引用したのか理由の説明がなく、批判者から「VSAを引用しておいてVSAと相反する結論を出している」と突っ込まれる可能性があります。

---

### 5. 冪等性チェックはInteractor層の責務か？という問い

`AnalyzeTaskInteractor`の例で、冪等性チェック（既に分析済みかどうか）をInteractor層に入れています。批判者からは「冪等性はドメインの関心事であってDomain層に書くべきでは？」という指摘が来る可能性があります。逆に「アプリケーション層の関心事で正しい」とする立場もあり、どちらが正解かの説明が記事中にありません。

---

### 6. 「テストが面倒にならない」の説明が不十分

テストの例としてHandlerのモックを示していますが、Interactor自体のユニットテストについての言及がありません。パススルーInteractorのテストは薄くなりがちで、「テストファイルの数が増えるコスト」や「何もしないテストの価値」については触れられていません。

1つ目は、`errors.Unwrap(err)` で取り出した値をログに出すと同じメッセージが重複して見えること。2つ目は、`HandleError` は `errors.As` で `*CodedError` を判定しているのに、なぜ `Unwrap()` が必要なのか記事中に説明がなく、読者が混乱します。`Unwrap()` を持たせる具体的なユースケースを示すか、不要なら削除すべきです。

### ⚠️ 重要な実装の欠落：`model.ErrTaskNotFound → apperror.ErrNotFound` の変換コードがない

「エラーの流れ全体像」の表では：

```
ErrTaskNotFound ──→ apperror.ErrNotFound ──→ 404 Not Found
（sentinel）       （CodedError変換）
```

と示されていますが、この変換を行うInteractorのコードが記事のどこにも登場しません。Repositoryが `model.ErrTaskNotFound` を返した後、誰がそれを `apperror.ErrNotFound` に変換するのかが宙に浮いています。「エラーの流れ」を解説する記事の核心部分だけに、ここが欠けているのは致命的な穴です。

---

### ⚠️ バリデーションエラーの情報が失われる

```go
if err := i.validateInput(req); err != nil {
    return nil, apperror.ErrInvalidFormat // CodedError
}
```

`validateInput` の結果にかかわらず常に `"invalid format"` という汎用メッセージだけが返ります。実際のAPIでは「どのフィールドが、なぜ不正か」をクライアントに返す必要があるケースがほとんどです。フィールドレベルのバリデーションエラーをどう扱うかについて触れられていないため、このコードをそのまま採用するとユーザー体験が悪くなります。

---

### ⚠️ `isDuplicateKeyError` が定義されていない

Interactorのコードで：

```go
if isDuplicateKeyError(err) {
    return apperror.ErrDuplicateEntry
}
```

と使われていますが、この関数の実装がどこにもありません。ORMやDBドライバに依存した実装になるため、実装例か少なくとも「ドライバ固有のエラーコードを確認する」という注記が必要です。

---

### ⚠️ 代替パターンの `nil` 返しの問題点に言及がない

新たに追加された「nilを返してInteractorで判断する」パターンは良い追加ですが、このパターンには既知の問題があります。

```go
return nil, nil // 見つからなかった事実だけを返す
```

`(nil, nil)` を返す設計はGoコミュニティで批判されることがあります。呼び出し側が `err == nil` だからといって成功とは限らず、`job == nil` も別途チェックが必要になり、チェック漏れのリスクがあります。この点への言及なしに「どちらも実用上は機能します」と結ぶのは不正確です。

---

### ⚠️ 「セキュリティの観点」でHTTPステータスの情報漏洩に触れていない

存在確認のエラーでの情報漏洩（メールアドレスの例）は言及されていますが、**HTTPステータスコード自体も情報漏洩の原因になりえる**点が抜けています。たとえば `404 Not Found` と `403 Forbidden` を使い分けることで、リソースの存在が推測できてしまうケース（例：`/admin/users/123` が `404` なら存在しない、`403` なら存在する）はセキュリティ上よく議論される問題です。

| `Unwrap()` の意図説明がない | ⭐⭐ 中程度 |

## 矛盾の整理

**「教科書の主張と実務のギャップ」セクション**では、このパターンを「合理的」として推奨しています：

```go
// Repository層でgorm固有エラー → ドメインsentinel errorに変換
if errors.Is(result.Error, gorm.ErrRecordNotFound) {
    return nil, model.ErrTaskNotFound  // ← ドメインのsentinel error
}
```

**「変換場所の判断基準」セクション**の判断表には：

| 条件 | 変換場所 | 例 |
|---|---|---|
| エラーの意味が1対1で対応する | **Repository層** | 「レコードなし」→「タスクが見つからない」 |

と書かれているので、ここまでは一貫しています。

ところが、その直後に示されている ✅ コード例は：

```go
// ✅ インフラ層は事実を返す（変換しない）
return fmt.Errorf("failed to save user: %w", result.Error)

// ✅ Interactor層でビジネス判断する
if isDuplicateKeyError(err) {
    return apperror.ErrDuplicateEntry
}
```

これは「**インフラ層は変換せず、Interactorで判断する**」パターンのコードであり、判断表の1行目（Repository層で変換）とは**逆のパターン**のコードです。

---

## 問題の構造

「変換場所の判断基準」セクションは、本来2つのパターンを表で整理した上で、それぞれの ✅ コード例を並べるべきでした。しかし実際には：

- 表の1行目（Repository層で変換）→ コード例なし
- 表の2行目（Interactor層で変換）→ ✅ コード例あり

という非対称な構成になっており、「変換すべきでないケース」のコードが再掲されているだけで、セクション冒頭で推奨した「Repository層での変換」のコードが示されていません。

読者は「変換場所の判断基準」セクションの ✅ コードだけ読むと、「インフラ層は変換せずにラップして返すのが正しい」と理解してしまいます。記事全体の主張と正反対の印象を与えるため、このセクションには `FindByID` のような Repository層での変換例も ✅ として明示的に追加するのが適切です。

# Smart Commit with Gitmoji

Execute the following steps non-interactively:

## Branch Management

- If currently on `main` or `master` branch, create and checkout a new feature branch with a descriptive name
- If on any other branch, proceed with commit on current branch (no branch creation)
- If no changes are detected, exit without doing anything

## Change Analysis & Commit

1. **Review all changes** using `git status` and `git diff --staged` (or `git diff` if nothing staged)
2. **Stage changes** if not already staged using `git add .` or selective staging
3. **Analyze the changes** to understand:
   - Type of changes (new feature, bug fix, refactor, etc.)
   - Scope of changes (which files/components affected)
   - Impact and purpose of the changes

## Commit Message Creation

Create a commit message following this format:

```
<gitmoji> <type>: <concise description>

<optional detailed explanation if needed>
```

### Gitmoji Selection Guide

Choose the most appropriate gitmoji from <https://gitmoji.dev/> based on change type:

**Common Patterns:**

- ✨ `:sparkles:` - New features
- 🐛 `:bug:` - Bug fixes
- 🎨 `:art:` - Code structure/format improvements
- ⚡️ `:zap:` - Performance improvements
- 📝 `:memo:` - Documentation updates
- 🔧 `:wrench:` - Configuration changes
- ♻️ `:recycle:` - Code refactoring
- 🚨 `:rotating_light:` - Fix linter warnings
- ✅ `:white_check_mark:` - Add/update tests
- 🔒️ `:lock:` - Security fixes
- ⬆️ `:arrow_up:` - Upgrade dependencies
- 🚚 `:truck:` - Move/rename files
- 🗑️ `:wastebasket:` - Remove deprecated code

### Message Guidelines

- **Title**: Keep under 50 characters, use imperative mood
- **Body**: Explain "what" and "why", not "how"
- **Language**: Write in English
- **Tone**: Professional and concise

## Final Steps

- Execute the commit with the generated message
- Display a summary of what was committed
- Suggest next steps (push to remote, create PR, etc.)
