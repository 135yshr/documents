## はじめに

:::message

本記事は「DDD / クリーンアーキテクチャ」連載の1つです。GoプロジェクトにDDDを導入して半年間運用した実体験をもとに、効果が大きかったプラクティスと、導入したものの期待ほど効果がなかったものを振り返ります。

:::

私のチームでは、約半年前にGoで開発している業務システムにDDD（ドメイン駆動設計）を導入しました。チームメンバーは5名で、プロダクトは受注管理を中心とした社内向けの業務アプリケーションです。

導入前は「DDDを入れれば設計が良くなるはず」という漠然とした期待がありましたが、半年運用してみると**効果が大きかったプラクティスと、導入コストに見合わなかったプラクティス**がはっきり分かれました。

この記事では、私のチームの実体験をもとに、DDDの各プラクティスを「やってよかった」「やらなくてよかった」に分類して共有します。あくまで1チームの事例ですが、DDD導入を検討している方の参考になれば嬉しいです。

---

## やってよかったこと

### 1. ユビキタス言語の整備

**効果：5/5（最大）**

DDDで最も効果が大きかったのは、間違いなくユビキタス言語の整備です。

導入前は、同じ概念をチーム内で別々の名前で呼んでいました。たとえば「注文」を表す用語が、コード上では`Order`、ドキュメントでは「受注」、会議では「案件」とバラバラでした。この不一致がバグの温床になっていました。

私のチームでは、以下の手順でユビキタス言語を整備しました。

1. **用語集をNotionに作成する**: ドメインの主要な概念を洗い出し、日本語名・英語名（コード上の名前）・定義を一覧化しました
2. **コード上の命名を統一する**: 変数名、型名、パッケージ名をユビキタス言語に合わせてリネームしました
3. **PRレビューで用語を確認する**: 新しい概念が登場したとき、用語集に追加してからコードを書くルールにしました

```go
// ❌ Before：用語がバラバラ
type Case struct {       // 「案件」
    ProjectID string     // 「プロジェクト」
    ClientName string    // 「取引先」
}

// ✅ After：ユビキタス言語に統一
type Order struct {      // 「受注」
    OrderID    string    // 「受注ID」
    CustomerName string  // 「顧客名」
}
```

用語が統一されたことで、ドメインエキスパート（業務担当者）との会話がスムーズになりました。「この`Case`って受注のことですか」といった確認が不要になり、打ち合わせの効率が上がりました。

### 2. 値オブジェクトの導入

**効果：4/5**

値オブジェクト（Value Object）は、導入コストが低いわりに効果が大きいプラクティスでした。

導入前は、金額や数量をすべて`int`や`string`で表現していました。型のレベルでは区別がつかず、金額と数量を取り違えるバグが発生していました。

```go
// ❌ Before：プリミティブ型でドメイン概念を表現
func CalcTotal(price int, quantity int, taxRate float64) int {
    return int(float64(price*quantity) * (1 + taxRate))
}

// 引数の順序を間違えてもコンパイルエラーにならない
total := CalcTotal(quantity, price, taxRate) // 逆になっているがコンパイル通る
```

値オブジェクトを導入すると、型のレベルでドメイン概念を区別できます。

```go
// ✅ After：値オブジェクトで型安全にする
type Money struct {
    amount int
}

func NewMoney(amount int) (Money, error) {
    if amount < 0 {
        return Money{}, fmt.Errorf("金額は0以上である必要があります: %d", amount)
    }
    return Money{amount: amount}, nil
}

func (m Money) Multiply(quantity Quantity) Money {
    return Money{amount: m.amount * quantity.Value()}
}

type Quantity struct {
    value int
}

func NewQuantity(value int) (Quantity, error) {
    if value <= 0 {
        return Quantity{}, fmt.Errorf("数量は1以上である必要があります: %d", value)
    }
    return Quantity{value: value}, nil
}
```

値オブジェクトを導入して特に良かった点は以下の3つです。

- **コンパイル時に型の取り違えを検出できる**: `Money`と`Quantity`は別の型なので、引数を間違えるとコンパイルエラーになります
- **バリデーションロジックが一箇所に集約される**: `NewMoney`で必ずバリデーションが実行されるため、不正な値が生成されません
- **ドメイン知識がコードに埋め込まれる**: 「金額は0以上」「数量は1以上」というビジネスルールが型に表現されます

### 3. 集約（Aggregate）によるトランザクション境界の明確化

**効果：4/5**

集約の設計は、データ整合性の問題を大幅に減らしました。

導入前は関連するエンティティをバラバラに更新しており、一部だけ更新されて不整合になるケースがありました。

```go
// ❌ Before：注文と注文明細をバラバラに更新
func (s *OrderService) AddItem(orderID string, item Item) error {
    order, _ := s.orderRepo.FindByID(orderID)
    order.Items = append(order.Items, item)
    s.orderRepo.Save(order)           // ここで失敗すると…

    order.TotalAmount += item.Price    // 不整合な状態になる
    s.orderRepo.Save(order)
    return nil
}
```

集約として設計し直すと、注文とその明細は常にまとめて操作されます。

```go
// ✅ After：集約ルートが整合性を保証する
type Order struct {
    id          OrderID
    items       []OrderItem
    totalAmount Money
    status      OrderStatus
}

func (o *Order) AddItem(product Product, qty Quantity) error {
    if o.status != StatusDraft {
        return fmt.Errorf("確定済みの受注には明細を追加できません")
    }
    item := NewOrderItem(product, qty)
    o.items = append(o.items, item)
    o.recalcTotal()
    return nil
}

func (o *Order) recalcTotal() {
    total := NewMoneyZero()
    for _, item := range o.items {
        total = total.Add(item.Subtotal())
    }
    o.totalAmount = total
}
```

集約ルート（`Order`）を経由してのみ状態を変更するため、`totalAmount`と`items`の不整合が構造的に発生しません。

### 4. リポジトリパターンによるDB実装の分離

**効果：3/5**

リポジトリパターンは、テスタビリティの向上に貢献しました。usecase層のテストでDBを起動する必要がなくなり、テストの実行速度が大幅に改善しました。

```go
// domain/repository/order_repository.go
type OrderReader interface {
    FindByID(ctx context.Context, id OrderID) (*Order, error)
}

type OrderWriter interface {
    Save(ctx context.Context, order *Order) error
}
```

```go
// usecase/create_order_test.go
type mockOrderWriter struct {
    saved *domain.Order
}

func (m *mockOrderWriter) Save(ctx context.Context, order *domain.Order) error {
    m.saved = order
    return nil
}

func TestCreateOrder(t *testing.T) {
    writer := &mockOrderWriter{}
    interactor := NewCreateOrderInteractor(writer)

    err := interactor.Execute(ctx, input)

    assert.NoError(t, err)
    assert.Equal(t, "customer-1", writer.saved.CustomerID())
}
```

## やらなくてよかったこと

### 1. ドメインイベントの早期導入

**効果：2/5（導入コストに見合わなかった）**

DDDの書籍で推奨されていたため、初期段階からドメインイベントを導入しました。しかし、私のチームの規模とプロダクトの特性には合いませんでした。

```go
// 導入したがほとんど使われなかったドメインイベント
type OrderCreatedEvent struct {
    OrderID   string
    CreatedAt time.Time
}

type EventPublisher interface {
    Publish(ctx context.Context, event DomainEvent) error
}
```

問題は以下の点でした。

- **イベントの購読者がいない**: マイクロサービス間の連携がなく、イベントを受け取るサービスも存在しませんでした
- **イベント基盤の運用コスト**: メッセージキュー（Amazon SQS等）のインフラ構築・運用が必要になり、チームの負担が増えました
- **デバッグが困難だった**: イベント駆動にした結果、処理の流れが追いづらく、障害調査の時間が増えました

結局、ドメインイベントは半年後に撤去し、**直接メソッド呼び出し**に戻しました。マイクロサービス化が進んだ段階で再導入する予定です。

### 2. 仕様パターン（Specification）の汎用実装

**効果：1/5（ほぼ使わなかった）**

Eric Evansの書籍に出てくる仕様パターンを汎用的に実装しました。

```go
// 汎用的な仕様パターン
type Specification[T any] interface {
    IsSatisfiedBy(entity T) bool
}

type AndSpecification[T any] struct {
    specs []Specification[T]
}

func (s *AndSpecification[T]) IsSatisfiedBy(entity T) bool {
    for _, spec := range s.specs {
        if !spec.IsSatisfiedBy(entity) {
            return false
        }
    }
    return true
}
```

この汎用実装は使いませんでした。理由は以下の通りです。

- **Goには関数型のアプローチが合う**: ジェネリクスで仕様パターンを実装するより、`func(Order) bool`のようなフィルタ関数を渡す方がGoらしく、コードも読みやすいです
- **実際の検索はSQLで行う**: メモリ上でのフィルタリングより、SQLのWHERE句で絞り込む方が圧倒的にパフォーマンスが良いです。仕様パターンは結局SQLに変換する必要があり、二重管理になりました

最終的に、フィルタリングが必要な場面では単純な関数を使う形に落ち着きました。

```go
// ✅ Goらしいフィルタリング
type OrderFilter struct {
    Status     *OrderStatus
    CustomerID *string
    CreatedFrom *time.Time
    CreatedTo   *time.Time
}

func (r *orderRepository) FindAll(ctx context.Context, filter OrderFilter) ([]*Order, error) {
    query := "SELECT * FROM orders WHERE 1=1"
    var args []interface{}

    if filter.Status != nil {
        query += " AND status = ?"
        args = append(args, *filter.Status)
    }
    // ...
}
```

### 3. 過度に厳格なレイヤー分離

**効果：2/5（開発速度を落とした）**

初期段階で4層のレイヤー分離を厳格に適用した結果、CRUDが主体の画面で不必要なボイラープレートが大量に発生しました。

```text
# 1つのCRUD画面に必要なファイル数：8ファイル
domain/model/product.go
domain/repository/product_repository.go
usecase/port/output/product_port.go
usecase/create_product_interactor.go
usecase/get_product_interactor.go
usecase/update_product_interactor.go
interface/rest/handler/product_handler.go
infrastructure/postgres/product_repository.go
```

単純なCRUDのために8ファイルを作成・メンテナンスするのは、明らかに過剰でした。私のチームでは、以下の基準でレイヤー分離の粒度を調整しました。

| ドメインの複雑さ         | 構成                                           |
| ------------------------ | ---------------------------------------------- |
| 単純なCRUD               | handler → repository（2層、usecaseを省略）     |
| ビジネスルールが少しある | handler → usecase → repository（3層）          |
| 複雑なビジネスルール     | handler → usecase → domain → repository（4層） |

**すべてを同じ構成にする必要はありません。** ドメインの複雑さに応じてレイヤーの数を変えることで、開発速度と設計品質のバランスが取れました。

## チームへの浸透で苦労した点と工夫

### 苦労した点1：「なぜ今のやり方ではダメなのか」が伝わらない

DDDの導入を提案したとき、チームメンバーの反応は「今のコードでも動いているのに、なぜ変える必要があるのか」でした。これは当然の反応です。

**工夫**: 実際のバグ事例を使って説明しました。「先月の受注金額バグは、`int`型で金額と数量を取り違えたのが原因でした。値オブジェクトを使えば、型レベルで防げます」という具体例を示すと、理解が得られました。

### 苦労した点2：DDDの用語が難しい

「ユビキタス言語」「集約」「値オブジェクト」「境界づけられたコンテキスト」…DDDの用語はチームメンバーにとって馴染みがありませんでした。

**工夫**: DDDの用語をそのまま使わず、チーム内で分かりやすい言い換えを用意しました。

| DDDの用語                  | チーム内での呼び方   |
| -------------------------- | -------------------- |
| ユビキタス言語             | 用語辞書             |
| 値オブジェクト             | 型付きの値           |
| 集約                       | まとめて保存する単位 |
| リポジトリ                 | データアクセス層     |
| 境界づけられたコンテキスト | サービスの担当範囲   |

DDDの概念を理解してもらった後で、正式な用語を紹介するようにしました。

### 苦労した点3：全部を一度に導入しようとした

最初は「DDDのプラクティスを全部導入しよう」と意気込んでいました。しかし、チーム全員の理解が追いつかず、混乱を招きました。

**工夫**: 段階的に導入するスケジュールに変更しました。

```text
1ヶ月目：ユビキタス言語の整備 + 用語集の作成
2ヶ月目：値オブジェクトの導入（新規コードから適用）
3ヶ月目：集約の設計（コアドメインのみ）
4ヶ月目：リポジトリパターンの導入
5〜6ヶ月目：既存コードのリファクタリング
```

1ヶ月に1つのプラクティスを導入し、チームが慣れてから次に進む方式にしたことで、混乱が減りました。

## 「次にやるなら」の改善ポイント

半年間の経験を踏まえて、もしDDD導入をやり直すとしたら変えたい点をまとめます。

### 1. コアドメインの特定を最初にやる

私のチームでは、すべてのモジュールに同じレベルのDDDを適用しようとしました。しかし、DDDの効果が大きいのは**ビジネスルールが複雑な領域（コアドメイン）**だけです。

次にやるなら、最初にコアドメインとサポーティングドメインを分類し、コアドメインにのみDDDを本格的に適用します。

```text
コアドメイン（DDD本格適用）：
  - 受注管理（複雑なビジネスルール、金額計算、ステータス遷移）

サポーティングドメイン（軽量な構成）：
  - ユーザー管理（CRUD中心）
  - 通知（外部サービス連携中心）
  - マスタ管理（参照データ中心）
```

### 2. モブプログラミングで最初のモデリングをする

最初の集約設計を私一人で行い、チームにレビューしてもらう形を取りました。しかし、設計の背景や意図が十分に共有されず、「なぜこうなっているのか」という質問が繰り返し発生しました。

次にやるなら、**最初の1〜2つの集約はモブプログラミングによりチーム全員で設計します**。時間はかかりますが、設計の意図が全員に共有されるため、その後の開発速度が上がります。

### 3. 戦術的パターンより先に戦略的パターンを導入する

私のチームでは、値オブジェクトや集約といった**戦術的パターン**から導入しました。しかし、今振り返ると、**戦略的パターン**（境界づけられたコンテキスト、コンテキストマップ）を先に整理すべきでした。

戦略的パターンでシステムの全体像を把握してからでないと、「どこにDDDを適用すべきか」「どこはCRUDで十分か」の判断ができません。

## まとめ

DDDを半年間運用した結果を振り返ると、以下のようにまとまります。

### 効果が大きかったプラクティス

| プラクティス       | 効果 | 理由                                   |
| ------------------ | ---- | -------------------------------------- |
| ユビキタス言語     | 5/5  | チーム内の認識のズレが激減した         |
| 値オブジェクト     | 4/5  | 型レベルでバグを防止できた             |
| 集約               | 4/5  | データ整合性の問題が構造的に解消された |
| リポジトリパターン | 3/5  | テスタビリティが大幅に向上した         |

### 効果が薄かったプラクティス

| プラクティス       | 効果 | 理由                                               |
| ------------------ | ---- | -------------------------------------------------- |
| ドメインイベント   | 2/5  | イベントの購読者がおらず、インフラコストだけ増えた |
| 仕様パターン       | 1/5  | Goの関数型アプローチとSQLの方が実用的だった        |
| 過度なレイヤー分離 | 2/5  | CRUDにはボイラープレートが過剰だった               |

DDDは「全部入り」で導入するものではありません。プロジェクトの規模・チームの成熟度・ドメインの複雑さに応じて、**必要なプラクティスを選んで段階的に導入する**のが成功の鍵だと実感しました。

まずはユビキタス言語と値オブジェクトから始めてみてください。この2つだけでも、コードの品質は確実に向上します。
