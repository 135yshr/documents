## はじめに

セキュリティレビューで指摘を受けたとき、すべてを修正するのが正解とは限りません。もちろん、本物の脆弱性を放置してよいという話ではありません。しかし、指摘の中には「脆弱性」ではなく「設計上の特性」であるものが混ざっていることがあります。それを見極める力は、OSS メンテナやセキュリティに関わる開発者にとって重要なスキルです。

この記事では、筆者が自作のトランスパイラ（Go で書かれたコード生成ツール）のランタイムについて上記の指摘を受けた実体験をもとに、「修正しない」と判断した思考過程を解説します。

## 指摘された内容

セキュリティレビューで報告されたのは、コード生成ツールのランタイムライブラリが提供する以下の機能についてでした。

### ファイル I/O

- 任意のパスからファイルを読み取れる
- パストラバーサル制限なし（`../../etc/passwd` のようなパスも読める）

### HTTP 通信

- 任意のホストに HTTP リクエストを送信できる
- SSRF（Server-Side Request Forgery）対策なし
- 内部ネットワークのサービスにもアクセス可能

一見すると深刻な脆弱性に見えます。パストラバーサルや SSRF は、Web アプリケーションのセキュリティでは定番の脅威であり、実際に CVE が発行されることも多い問題です。この指摘をそのまま受け取れば、「すぐに修正すべき重大な脆弱性」という結論になるでしょう。

しかし、結論から言えば、これは修正しませんでした。その理由を順に説明します。

## なぜ「修正しない」と判断したのか

### 理由 1: コンパイル済みバイナリの立場を考える

このツールはトランスパイラです。ソースコードを Go に変換し、`go build`
でネイティブバイナリを生成します。生成されたバイナリは OS 上で直接動作するプログラムです。

この「OS 上で直接動作するプログラム」という立場は、Go、Python、Ruby、Rust のコンパイラやインタプリタで作られたプログラムと全く同じです。

- `go run main.go` で実行されるプログラムも任意のファイルにアクセスできる
- `python script.py` で実行されるプログラムも任意の HTTP リクエストを送信できる
- `rustc` でコンパイルされたバイナリにもパストラバーサル制限はない

これらのランタイムに「パストラバーサル制限がない」「SSRF 対策がない」という CVE は報告されていません。それは当然で、汎用プログラミング言語のランタイムがファイルやネットワークへのアクセスを提供するのは、脆弱性ではなく基本機能だからです。

自作のトランスパイラも、規模は小さいとはいえ同じカテゴリのソフトウェアです。同種のソフトウェアで問題とされていない動作を、自分のツールだけ「脆弱性」として扱う合理的な理由はありません。

### 理由 2: Threat Model の範囲

セキュリティの判断で最も重要なのは、Threat
Model（脅威モデル）を明確にすることです。「何を守るのか」「何がスコープ内なのか」を定義しないと、すべての指摘が等しく重要に見えてしまいます。

このツールの Threat Model を整理すると、以下のようになります。

| 脅威                                                  | スコープ内か | 理由                                 |
| ----------------------------------------------------- | ------------ | ------------------------------------ |
| コンパイラ自体への入力攻撃（悪意ある .nyan ファイル） | はい         | コンパイラはユーザーの入力を処理する |
| コンパイル済みバイナリの OS レベルの動作              | いいえ       | OS/コンテナ層の責務                  |
| ランタイムライブラリの API 設計                       | いいえ       | 汎用言語として標準的な設計           |

コンパイラ自体がバッファオーバーフローやコマンドインジェクションの脆弱性を持っていたら、それは間違いなく修正すべきです。しかし、コンパイルされたプログラムが OS レベルの機能にアクセスできることは、コンパイラの脅威モデルの範囲外です。

### 理由 3: サンドボックスは別レイヤーの責務

「信頼できないコードから任意のファイルにアクセスできて危険ではないか」という疑問は正当です。しかし、その対策を担うのは言語ランタイムでなく、OS やインフラのレイヤーです。

信頼できないコードの実行を制限したい場合には、以下のような手段があります。

- **コンテナ（Docker）** でファイルシステムを隔離する
- **seccomp プロファイル** で syscall を制限する
- **ファイアウォールルール** でネットワークアクセスを制限する
- **AppArmor / SELinux** でファイルアクセスを細かく制御する

これらは、どの言語で書かれたプログラムに対しても共通に適用できる汎用的な仕組みです。言語ランタイムに独自のサンドボックスを実装すると、OS 側の制御と二重管理になるだけでなく、中途半端な実装がかえってセキュリティ上の問題を生む可能性もあります。

もちろん、Deno のようにランタイムレベルでパーミッションモデルを提供する言語もあります。しかし、それは言語の設計思想としての選択であり、すべての言語が採用すべき標準ではありません。Go、Python、Ruby、Rust がパーミッションモデルを持たないことは脆弱性とはみなされていません。

## 「修正しない」と判断したら何をするか

「修正しない」と判断しても、何もしなくてよいわけではありません。コード修正は行いませんが、以下の対応は必ず行うべきです。

### 1. SECURITY.md に設計意図を明記する

プロジェクトの SECURITY.md に、この動作が意図的な設計であることを明記します。

```markdown
## Runtime Capabilities

### File I/O and Network Access

Programs compiled with this tool run as native executables with the same OS-level privileges as any
Go, Python, or Rust program. The runtime library provides file I/O and HTTP capabilities without
built-in sandboxing.

**This is by design.** Like other compiled languages, access control is the responsibility of the
execution environment, not the language runtime.

If you need to restrict a program's capabilities:

- Use containers (Docker) for filesystem isolation
- Apply seccomp profiles to limit system calls
- Configure firewall rules for network restrictions
- Use OS-level MAC (AppArmor, SELinux) for fine-grained control
```

こうすることで、将来同じ指摘を受けたときに「既に検討済みである」と示せます。ユーザーへの代替手段の案内にもなります。

### 2. 判断根拠を記録する

ADR（Architecture Decision
Record）や Issue のコメントに、判断根拠を記録しておきます。記録すべき内容は以下の 4 点です。

- **何が指摘されたか** -- 指摘の具体的な内容を正確に記録する
- **なぜ修正しないと判断したか** -- 技術的な根拠を明確に書く
- **どのような代替策があるか** -- ユーザーが取れる対策を列挙する
- **この判断を見直すべきトリガー条件**
  -- 例えば「WASM 環境でのサンドボックス実行をサポートする場合は再検討する」など

判断根拠の記録は、未来の自分やチームメンバーに向けたものです。数か月後に同じ議論が繰り返されるのを防ぎ、プロジェクトの意思決定の一貫性を保つ役割を果たします。

## 「修正しない」判断のチェックリスト

最後に、「修正しない」という判断が妥当かどうかを検証するためのチェックリストを紹介します。以下の質問にすべて「はい」と答えられるなら、その判断は妥当と言えるでしょう。

1. **同種のソフトウェアでも同じ動作か** --
   Go の標準ライブラリで同じ機能を使っても同じリスクが存在する。他の言語やツールでも同様の設計になっている
2. **Threat Model のスコープ外か**
   -- この脅威は別のレイヤー（OS、コンテナ、ネットワーク）で対処すべきものである
3. **設計意図が文書化されているか** --
   SECURITY.md や ADR に判断根拠が記録されている。将来同じ指摘があったときに参照できる
4. **ユーザーに代替手段があるか** --
   OS 側の仕組み（コンテナ、seccomp、ファイアウォールなど）で軽減できる

1 つでも「いいえ」がある場合は、修正を検討するか、少なくとも追加の文書化や対応が必要です。

## アンチパターン: 修正しないことの悪い理由

逆に、以下のような理由で「修正しない」と判断するのは危険です。これらは正当な根拠にはなりません。

- **「攻撃される可能性が低いから」**
  -- 可能性の低さは対策しない理由にならない。セキュリティでは「可能性が低いが影響が大きい」脅威こそ重要である
- **「修正が面倒だから」**
  -- 工数の問題は別途議論すべきだ。修正が困難であること自体は、修正しない技術的な根拠にならない
- **「誰も報告していないから」**
  -- 報告がないことは安全の証拠ではない。単に発見されていないだけ、あるいは報告せずに悪用されている可能性がある

これらの理由で修正を見送ると、後から実際に攻撃された際に「なぜ放置したのか」を説明できなくなります。「修正しない」判断をするなら、技術的に正当な根拠を持つべきです。

## まとめ

セキュリティレビューの指摘に対して「修正しない」と判断することは、適切な根拠があれば立派なセキュリティ対応の一形態です。

重要なポイントを振り返ります。

- **「修正しない判断」にこそ明確な根拠を残す**
  -- 曖昧な判断は後から問題になる。技術的な根拠の文書化が重要である
- **Threat Model を明確にする** -- 脅威モデルの定義により、スコープ内外の脅威を客観的に判断できる
- **「意図的な設計上の特性」と宣言することもセキュリティ対応である**
  -- 設計意図の明記により、脆弱性と設計上の特性を区別できるようになる
- **判断を記録する場所を持つ** --
  SECURITY.md、ADR、Issue コメントなど、チームや将来の自分が参照できる場所に残す

すべての指摘を機械的に修正するのではなく、Threat
Model に基づいて冷静に判断する。その姿勢こそが、信頼されるソフトウェアを作るための土台になるはずです。
