## はじめに

`regexp.Compile(userInput)` が危険なことは多くの開発者が知っている。しかし `fmt.Sprintf("^%s$", name)` のように正規表現パターンを組み立てるケースでは、エスケープを忘れがちだ。

筆者が Go で書かれたコード生成ツールのテストランナーをレビューした際、まさにこのパターンを発見した。関数名のフィルタリングで正規表現へ埋め込んでいたが、`regexp.QuoteMeta` は使われていなかった。本記事ではこの問題の影響と修正方法を解説する。

## 問題のコード: テスト関数名をフィルタリングする

以下は、テストランナーが特定の fuzz テスト関数だけを実行するコードだ。

```go
func runFuzzTest(name string) error {
    cmd := exec.Command("go", "test",
        fmt.Sprintf("-fuzz=^%s$", name),
        "-fuzztime=30s",
    )
    return cmd.Run()
}
```

一見問題なさそうだが、`name` に正規表現のメタ文字が含まれると予期しない動作になる。

## 何が起きるか

`name` にさまざまなメタ文字が含まれた場合の動作を見てみよう。

| 入力 `name`   | 生成されるパターン | 結果                       |
| ------------- | ------------------ | -------------------------- |
| `fuzz_add`    | `^fuzz_add$`       | 意図通り1つにマッチ        |
| `fuzz_.*`     | `^fuzz_.*$`        | 全 fuzz テストにマッチ     |
| `fuzz_(a\|b)` | `^fuzz_(a\|b)$`    | 2つのテストにマッチ        |
| `fuzz_[abc]`  | `^fuzz_[abc]$`     | 文字クラスとして解釈される |
| `fuzz_a]b`    | `^fuzz_a]b$`       | 正規表現構文エラー         |

`_` が正規表現では「任意の1文字」を意味する `.` ではないことに注意。ここでの問題は `_` ではなく、`.*` や `()`, `[]` といった明確なメタ文字だ。

## セキュリティへの影響

単なるバグと思われがちだが、セキュリティ上の影響がある。

**意図しないテスト実行**: `fuzz_.*` のようなパターンで全 fuzz テストが同時に実行され、CI リソースを大量に消費する。fuzz テストは長時間動作するため、DoS 的な影響を与えうる。

**テストのバイパス**: パターンの構文エラーにより、テストコマンド自体が失敗する。エラーハンドリングの不備でテストがスキップされる可能性もある。

**外部入力が来る場合**: 設定ファイルや API からテスト関数名を受け取るケースでは、攻撃者が正規表現メタ文字を意図的に注入できる。

## 修正: `regexp.QuoteMeta` を使う

修正はシンプルだ。`regexp.QuoteMeta` でメタ文字をエスケープする。

```go
func runFuzzTest(name string) error {
    cmd := exec.Command("go", "test",
        fmt.Sprintf("-fuzz=^%s$", regexp.QuoteMeta(name)),
        "-fuzztime=30s",
    )
    return cmd.Run()
}
```

`regexp.QuoteMeta` はすべてのメタ文字の前にバックスラッシュを付加する。

```go
regexp.QuoteMeta("fuzz_.*")     // => "fuzz_\.\*"
regexp.QuoteMeta("fuzz_(a|b)")  // => "fuzz_\(a\|b\)"
regexp.QuoteMeta("fuzz_[abc]")  // => "fuzz_\[abc\]"
```

これにより、メタ文字がリテラル文字として扱われ、意図した関数名だけにマッチするようになる。

## よくあるパターン: QuoteMeta が必要な場面

`-fuzz` フラグ以外にも、同じ問題の発生しやすいパターンがある。

```go
// パターン1: テスト名のフィルタリング
fmt.Sprintf("-run=^%s$", regexp.QuoteMeta(testName))

// パターン2: ファイル名でのグレップ
pattern := fmt.Sprintf(`%s:\d+:`, regexp.QuoteMeta(filename))

// パターン3: ユーザー入力を含む検索
pattern := fmt.Sprintf(`\b%s\b`, regexp.QuoteMeta(searchTerm))
```

いずれのケースでも、外部から来る文字列を正規表現パターンに埋め込む際には `QuoteMeta` が必要だ。

## 各言語のエスケープ関数

この問題は Go に限らない。各言語のエスケープ関数を把握しておこう。

| 言語       | 関数                 |
| ---------- | -------------------- |
| Go         | `regexp.QuoteMeta()` |
| Python     | `re.escape()`        |
| Ruby       | `Regexp.escape()`    |
| Rust       | `regex::escape()`    |
| Java       | `Pattern.quote()`    |
| JavaScript | 標準関数なし         |

JavaScript には標準のエスケープ関数がないことで有名だ。手動で実装する場合は以下のようになる。

```js
function escapeRegExp(str) {
  return str.replace(/[.*+?^${}()|[\]\\]/g, "\\$&");
}
```

TC39 で `RegExp.escape()` の提案が進行中だが、2026年3月時点ではまだ標準化されていない。

## QuoteMeta が不要な場面

すべてに `QuoteMeta` を付ければいいわけではない。

- **正規表現として意図的に使いたい場合**: ユーザーが正規表現を入力する検索機能では、エスケープしてしまうと正規表現の機能が使えなくなる
- **リテラル文字のみで構成されることが保証されている場合**: UUID やハッシュ値など、英数字とハイフンしか含まないことが確実な値
- **静的に定義されたパターン**: `regexp.MustCompile` で定数パターンを定義する場合

判断基準は「この文字列にメタ文字が含まれうるか」だ。少しでも可能性があればデフォルトでエスケープし、意図的にメタ文字を利用したい場面のみエスケープしない。

## まとめ

- 正規表現パターンに外部文字列を埋め込むときは、`regexp.QuoteMeta`（または各言語の同等関数）を使う
- 「テスト名」「ファイル名」「ユーザー入力」など、メタ文字が混入する可能性のある値には特に注意が必要だ
- コードレビューで `fmt.Sprintf` と正規表現パターンの組み合わせを見つけたら、エスケープの有無をチェックしよう
- 迷ったらデフォルトは「エスケープする」側に倒す。エスケープが不要なケースを理解した上で、意図的に外す方が安全だ
