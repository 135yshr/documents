DDDで設計されたプロジェクトにCQRSを導入しようとすると、さまざまな疑問が出てきます。「ユースケースとCommandの違いは何か」「QueryServiceはどの層に置くのか」「リポジトリはどうなるのか」といった点です。

この記事では、DDD × CQRSの導入時に**誤解しやすいポイント**を整理し、**理想的なディレクトリ構成とコードサンプル**をもとに解説します。

---

## CQRSとは何か（1分でおさらい）

**Command Query Responsibility Segregation（コマンドクエリ責務分離）**

> CQRSの核心は「情報を更新するモデルと、情報を読み取るモデルを分けられる」という考え方です。ただし、**多くのシステムでCQRSは不必要なリスクと複雑性を加える**ことにも注意が必要です。
>
> — Martin Fowler, [CQRS](https://martinfowler.com/bliki/CQRS.html)

| 種別    | 役割                 | 状態変化 |
| ------- | -------------------- | -------- |
| Command | データを変更する処理 | あり     |
| Query   | データを読み取る処理 | なし     |

この2つの経路を**明確に分離する**アーキテクチャパターンです。

:::message alert

CQRSはすべてのシステムに適用すべきパターンではありません。Martin Fowlerも「複雑なドメインを持つ一部の状況でのみ有効」と述べています。小規模なCRUDアプリケーションへの導入は、逆に複雑性を高める可能性があります。

:::

## CQRSのルーツ：CQS原則

CQRSはBertrand Meyerが提唱した**CQS（Command Query Separation）原則**をアーキテクチャレベルに昇華させたものです。

> CQSの基本的な考え方は、オブジェクトのメソッドを2つのカテゴリに明確に分類することです。
>
> - **Query**：結果を返し、システムの状態を変更しない（副作用なし）
> - **Command**：システムの状態を変更するが、値を返さない
>
> — Martin Fowler, [CommandQuerySeparation](https://martinfowler.com/bliki/CommandQuerySeparation.html)

## 誤解1：「CQRSを導入するとユースケースがなくなる」

### ❌ よくある誤解

```text
CQRSを入れる
  → ユースケースをCommandとQueryに置き換える
  → UseCaseクラスは不要になる
```

### ✅ 正しい理解

**ユースケースという概念はなくならない。** ユースケースを「書き込み系（Command）」か「読み込み系（Query）」かに**振り分けるだけ**です。

```text
業務要件（ユースケース）
        ↓ 分類する

  データを変える？
  ┌──────┴──────┐
  Yes            No
  ↓              ↓
Command        Query
（注文する）   （注文一覧を見る）
```

ユースケースが先にあって、CQRSはその**実装方針の話**です。

## 誤解2：「ユースケースの名前をCommand/Queryに変える必要がある」

### ❌ よくある誤解（命名）

```text
PlaceOrderUseCase.ts → PlaceOrderCommand.ts に名前を変える
GetOrderListUseCase.ts → GetOrderListQuery.ts に名前を変える
```

### ✅ 正しい理解（命名）

名前はどちらでも問題ありません。**本質は名前ではなく、中の実装ルール**です。

```typescript
// 名前はUseCaseのままでもCQRSとして正しい
class GetOrderListUseCase {
  execute(): OrderListDTO[] {
    // ← Domainモデルを経由しない（Query的な実装）
    return this.orderQueryService.findAll();
  }
}
```

**重要なのは、チームで命名規則を統一すること。** 混在が一番の混乱を生みます。

```text
❌ 混在している状態（最悪）
├── PlaceOrderUseCase.ts   // なんでUseCaseなの？
├── CancelOrderCommand.ts  // なんでCommandなの？
└── GetOrderListQuery.ts   // なんでQueryなの？

✅ UseCase に統一（既存プロジェクトへの導入に推奨）
├── PlaceOrderUseCase.ts
├── CancelOrderUseCase.ts
└── GetOrderListUseCase.ts

✅ Command/Query に統一（新規プロジェクトに推奨）
commands/
├── PlaceOrderCommand.ts
└── CancelOrderCommand.ts
queries/
├── GetOrderListQuery.ts
└── GetOrderDetailQuery.ts
```

## 誤解3：「Query側もRepositoryを使う」

DDDにおけるRepositoryは、**集約（Aggregate）の永続化・復元**のために存在します。

> Repositoryは集約ルートのみに対して提供し、クライアントをドメインモデルに集中させ、オブジェクトのストレージとアクセスをRepositoryに委譲する。
>
> — Eric Evans, [Domain-Driven Design（2003）p.151](https://fabiofumarola.github.io/nosql/readingMaterial/Evans03.pdf)

### ❌ よくある実装

```typescript
// QueryでもRepositoryを使って集約を取得し、DTOに詰め替える
class GetOrderListUseCase {
  execute(): OrderListDTO[] {
    const orders = this.orderRepository.findAll(); // ← 集約を全件取得
    return orders.map((order) => ({
      // ← DTOに詰め替え
      orderId: order.id.value,
      customerName: ???, // ← 別集約の情報が取れない...
      totalAmount: order.totalAmount,
    }));
  }
}
```

これではN+1問題が起きたり、集約の境界をまたいだデータが取れなかったりします。

### ✅ CQRSの正しいアプローチ

**Repository（集約を返す）** と **QueryService（DTOを返す）** を使い分けます。

```typescript
// ✅ Command側：Repositoryを使う（従来通り）
class PlaceOrderUseCase {
  execute(command: PlaceOrderInput): void {
    const order = Order.place(command.items, command.customerId);
    this.orderRepository.save(order); // ← 集約の永続化
  }
}

// ✅ Query側：QueryServiceを使う（CQRSの新しいやり方）
class GetOrderListUseCase {
  execute(): OrderListDTO[] {
    return this.orderQueryService.findAll(); // ← DTOを直接返す
  }
}
```

|            | Repository          | QueryService               |
| ---------- | ------------------- | -------------------------- |
| 返すもの   | 集約（DomainModel） | DTO（画面用データ）        |
| 役割       | 集約の保存・復元    | 画面に必要な形でデータ取得 |
| JOINの制約 | 集約の境界を守る    | 自由にJOINしてOK           |
| DDDの制約  | 受ける              | 受けない                   |

## 誤解4：「QueryServiceはインフラ層に置く」

### ❌ よくある誤解（配置）

```text
QueryServiceはSQLを書くのでインフラ層に置く
  → アプリケーション層からインフラ層を直接参照する
  → 依存関係が壊れる 🚨
```

### ✅ 正しい理解（配置）

これはRepositoryパターンとまったく同じ構造です。

```text
【Repository】                    【QueryService】

application/                      application/
└── IOrderRepository.ts           └── IOrderQueryService.ts
      （インターフェース）                （インターフェース定義）

infrastructure/                   infrastructure/
└── MySQLOrderRepository.ts       └── MySQLOrderQueryService.ts
      （実装：SQLを書く）                （実装：SQLを書く）
```

インターフェースはアプリケーション層に、実装はインフラ層に。**SQLを書くクラスは必ずインフラ層**です。

```typescript
// ✅ application層：インターフェースを定義（SQLは知らない）
interface IOrderQueryService {
  findAll(): OrderListDTO[];
  findById(id: string): OrderDetailDTO | null;
}

// ✅ application層：インターフェースにだけ依存
class GetOrderListUseCase {
  constructor(
    private queryService: IOrderQueryService, // ← インターフェース
  ) {}

  execute(): OrderListDTO[] {
    return this.queryService.findAll();
  }
}

// ✅ infrastructure層：SQLの詳細はここだけ
class MySQLOrderQueryService implements IOrderQueryService {
  findAll(): OrderListDTO[] {
    return this.db.query(`
      SELECT o.id, c.name as customer_name, o.total, COUNT(i.id) as item_count
      FROM orders o
      JOIN customers c ON o.customer_id = c.id
      JOIN order_items i ON o.id = i.order_id
      GROUP BY o.id, c.name, o.total
    `);
  }
}
```

## 誤解5：「依存の方向はHandler→UseCase→Domain→Infraの一方向」

これは**依存性逆転の原則（DIP）** に関わる重要な誤解です。

> 最も柔軟なシステムとは、ソースコードの依存関係が具体的な実装ではなく、抽象（インターフェース）だけを参照するものである。
>
> — Robert C. Martin, Clean Architecture
>
> 高レベルモジュールは低レベルモジュールに依存してはならない。両者とも抽象に依存すべきである。
>
> — Robert C. Martin, Agile Software Development: Principles, Patterns, and Practices（[Wikipedia: Dependency inversion principle](https://en.wikipedia.org/wiki/Dependency_inversion_principle) より）

### ❌ よくある誤解（依存方向）

```text
Handler → UseCase → Domain → Infrastructure（一方向に依存）
```

### ✅ 正しい理解（依存方向）

**「呼び出しの流れ」と「依存の方向」は別物**です。

```text
【呼び出しの流れ】（直感的な理解として正しい）
Handler → UseCase → Domain → Infrastructure

【依存の方向】（Infraだけ逆転する）
Handler → UseCase → Domain ← Infrastructure
                      ↑
               インターフェースを定義
               Infrastructureが実装（依存性逆転）
```

この仕組みによって「MySQLをPostgreSQLに変えても、Domainは一切変更不要」という状態が実現できます。

## 理想的なディレクトリ構成

```text
src/
├── presentation/                    # ハンドラ層（Controller, Resolver等）
│   └── order/
│       └── OrderController.ts
│
├── application/                     # アプリケーション層
│   ├── usecases/                    # ユースケース（命名規則で統一）
│   │   ├── PlaceOrderUseCase.ts     # Command系（Domainを通す）
│   │   ├── CancelOrderUseCase.ts    # Command系
│   │   ├── GetOrderListUseCase.ts   # Query系（Domainを通さない）
│   │   └── GetOrderDetailUseCase.ts # Query系
│   │
│   └── query-services/              # QueryServiceのインターフェース ← ここに置く！
│       └── IOrderQueryService.ts
│
├── domain/                          # ドメイン層（DDDの核心）
│   └── order/
│       ├── Order.ts                 # 集約ルート
│       ├── OrderItem.ts             # エンティティ
│       ├── OrderId.ts               # 値オブジェクト
│       └── IOrderRepository.ts     # Repositoryインターフェース ← domain層に置く！
│
└── infrastructure/                  # インフラ層
    ├── repositories/                # Repository実装（集約の永続化）
    │   └── MySQLOrderRepository.ts
    └── query-services/              # QueryService実装（SQLはここだけ）
        └── MySQLOrderQueryService.ts
```

## コードサンプル全体像

### Domain層

```typescript
// domain/order/Order.ts（集約ルート）
export class Order {
  private constructor(
    public readonly id: OrderId,
    private customerId: CustomerId,
    private items: OrderItem[],
    private status: OrderStatus,
  ) {}

  static place(customerId: CustomerId, items: OrderItem[]): Order {
    if (items.length === 0) {
      throw new Error("注文には1つ以上の商品が必要です");
    }
    return new Order(OrderId.generate(), customerId, items, OrderStatus.PENDING);
  }

  cancel(): void {
    if (this.status !== OrderStatus.PENDING) {
      throw new Error("この注文はキャンセルできません");
    }
    this.status = OrderStatus.CANCELLED;
  }
}

// domain/order/IOrderRepository.ts（インターフェース）
export interface IOrderRepository {
  findById(id: OrderId): Promise<Order | null>;
  save(order: Order): Promise<void>;
}
```

### Application層

```typescript
// application/query-services/IOrderQueryService.ts（インターフェース）
export interface IOrderQueryService {
  findAll(): Promise<OrderListDTO[]>;
  findById(id: string): Promise<OrderDetailDTO | null>;
}

export type OrderListDTO = {
  orderId: string;
  customerName: string;
  totalAmount: number;
  itemCount: number;
  status: string;
  createdAt: Date;
};

// application/usecases/PlaceOrderUseCase.ts（Command系）
export class PlaceOrderUseCase {
  constructor(
    private readonly orderRepository: IOrderRepository, // ← Domain I/Fに依存
  ) {}

  async execute(input: PlaceOrderInput): Promise<void> {
    const order = Order.place(
      new CustomerId(input.customerId),
      input.items.map((i) => new OrderItem(i.productId, i.quantity, i.price)),
    );
    await this.orderRepository.save(order);
  }
}

// application/usecases/GetOrderListUseCase.ts（Query系）
export class GetOrderListUseCase {
  constructor(
    private readonly queryService: IOrderQueryService, // ← Application I/Fに依存
  ) {}

  async execute(): Promise<OrderListDTO[]> {
    // DomainModelを経由しない！
    return this.queryService.findAll();
  }
}
```

### Infrastructure層

```typescript
// infrastructure/repositories/MySQLOrderRepository.ts
export class MySQLOrderRepository implements IOrderRepository {
  async findById(id: OrderId): Promise<Order | null> {
    const row = await this.db.query("SELECT * FROM orders WHERE id = ?", [id.value]);
    if (!row) return null;
    return this.reconstruct(row); // ← 集約として復元
  }

  async save(order: Order): Promise<void> {
    await this.db.query(
      "INSERT INTO orders ...", // ← 集約の永続化
    );
  }
}

// infrastructure/query-services/MySQLOrderQueryService.ts
export class MySQLOrderQueryService implements IOrderQueryService {
  async findAll(): Promise<OrderListDTO[]> {
    // 自由にJOINしてDTOを直接返す！集約の制約を受けない
    const rows = await this.db.query(`
      SELECT
        o.id as orderId,
        c.name as customerName,
        o.total_amount as totalAmount,
        COUNT(i.id) as itemCount,
        o.status,
        o.created_at as createdAt
      FROM orders o
      JOIN customers c ON o.customer_id = c.id
      JOIN order_items i ON o.id = i.order_id
      GROUP BY o.id, c.name, o.total_amount, o.status, o.created_at
    `);
    return rows;
  }
}
```

## 依存関係のまとめ図

```text
Presentation
    ↓ 依存
Application（UseCase + I/F定義）
    ↓ 依存（Command系のみ）
Domain（集約 + Repository I/F）
    ↑ 依存（implements）
Infrastructure（Repository実装 + QueryService実装）
```

**ポイント：インフラ層だけが上位層のインターフェースを実装する（依存性逆転）**

---

## 段階的な導入推奨

一気に全部やらず、スモールスタートが鍵です。

```text
Step 1: UseCaseをCommand系・Query系に分類するだけ
        → コード変更なし、意識だけ変える

Step 2: Query側でRepositoryを使わず、QueryServiceを導入
        → DBは共通、Event Sourcingなし

Step 3: 必要になったらReadモデル用DBを分離（オプション）
        → パフォーマンス要件が出てきたとき
```

## まとめ

| 誤解                                  | 正しい理解                                            |
| ------------------------------------- | ----------------------------------------------------- |
| ユースケースがなくなる                | ユースケースの概念は残る。Command/Queryに分類するだけ |
| 名前をCommand/Queryに変える必要がある | チームで統一すればどちらでもOK。混在がNG              |
| Query側もRepositoryを使う             | QueryServiceを使い、DTOを直接返す                     |
| QueryServiceはインフラ層に置く        | I/Fはアプリケーション層、実装がインフラ層             |
| 依存は全て一方向                      | 呼び出しと依存の向きは別。Infraだけ依存が逆転する     |

**「DB分離」や「Event Sourcing」をセットにしない軽量CQRSから始めると、むしろ「どこに何を書くか」のルールが明確になり、チームの混乱が減ります。**
