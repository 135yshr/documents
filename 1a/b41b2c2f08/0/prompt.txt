# ドキュメントレビュー

引数で指定された Markdown 記事ファイルの文章品質をレビューし、問題を指摘したうえで自動修正する。

## 入力

 に対象ファイルのパスが渡される（例: `articles/978121945958ed.md`）。パスが渡されない場合は「対象ファイルを指定してください（例: /review-doc articles/xxx.md）」とだけ返して終了すること。

## レビュー手順

### Step 1: 記事を読み込む

対象ファイルを Read ツールで全文読み込むこと。

### Step 2: 以下の 4 観点でレビューする

#### 観点 A: 冗長・繰り返し

- 同じ主張が複数箇所で繰り返されていないか
- 削れる前置き・つなぎ表現はないか（「〜がある。それは〜である。」のような回りくどい表現）
- 1 文が長すぎないか（目安: 80 文字超は分割を検討）

#### 観点 B: 論理構成・流れ

- セクション間の論理的なつながりが自然か
- 見出し階層が適切か（h2 → h3 の飛び越しがないか）
- 結論が導入の問いに対応しているか

#### 観点 C: 読者価値密度

- 各セクションに具体例・数値・コード例など「読んで良かった」と思える情報があるか
- 抽象的な主張だけで終わっているセクションはないか
- 読者が次のアクションを取れる情報が含まれているか

#### 観点 D: 文体の統一

- 記事全体が敬語（ですます調）で統一されているか
- 箇条書き内で「である」調が混在していないか
- 見出し直下の地の文とリスト項目で文体が揺れていないか

### Step 3: 指摘を出力する

以下のフォーマットで指摘を一覧表示すること。

```markdown
## レビュー結果: <ファイル名>

### 観点 A: 冗長・繰り返し

- [ ] L<行番号>: <指摘内容>（<改善案の要約>）

### 観点 B: 論理構成・流れ

- [ ] L<行番号>: <指摘内容>（<改善案の要約>）

### 観点 C: 読者価値密度

- [ ] L<行番号>: <指摘内容>（<改善案の要約>）

### 観点 D: 文体の統一

- [ ] L<行番号>: <指摘内容>（<改善案の要約>）

### 総評

<全体の品質に関する 2〜3 行のコメント>
```

指摘がない観点は「問題なし」とだけ記載すること。

### Step 4: 自動修正を適用する

指摘のうち、意味を変えずに改善できるもの（冗長表現の削除、文の分割など）は Edit ツールで自動修正すること。修正を適用したら、修正箇所のサマリーを出力すること。

内容の追加や構成の大幅な変更が必要な指摘は、修正案を提示するにとどめ、自動修正しないこと。

### Step 5: lint を実行する

修正後に以下を順に実行し、エラーがあれば修正すること。

```bash
npx prettier --write <対象ファイル>
npx markdownlint-cli2 <対象ファイル>
npx textlint <対象ファイル>
```

## 制約

- フロントマター（title, emoji, topics, published など）は変更しないこと
- 技術用語・固有名詞はそのまま残すこと
- 修正は最小限にとどめ、筆者の文体を尊重すること
- published の値は変更しないこと

## レビュー結果

### 🔴 技術的に誤っている箇所

#### 1. `errors.Is` が使えないという説明が不正確

記事の以下の部分：

> 「逆に言えば、`errors.Is(err, apperror.ErrNotFound)`のような値での判定はできません。特定のエラー種別を区別したい場合は、`Code()`メソッドで判定してください。」

**これは誤りです。** `apperror.ErrNotFound` はパッケージ変数として定義された `*CodedError` のポインタです：

```go
var (
    ErrNotFound = newCodedError("RES001", "resource not found", http.StatusNotFound)
)
```

`errors.Is` はエラーチェーンを辿りながらポインタ等値比較を行います。Interactorがこのポインタをそのまま返している（`fmt.Errorf`でラップしていない）ため、`errors.Is(err, apperror.ErrNotFound)` は **正しく `true` を返します**。

これは記事内の `errors.Is(err, model.ErrTaskNotFound)` と**まったく同じパターン**です。`ErrTaskNotFound` も `ErrNotFound` も、どちらもパッケージレベルのポインタ変数として定義されているため、`errors.Is` での判定は等しく機能します。

---

### 🟡 設計上の内部矛盾（ミスリーディング）

#### 2. 「Interactorがgormパッケージを知らずに済む」と述べながら、Interactorに `pgconn.PgError` が登場

記事はRepository層でエラー変換する利点として：

> 「Interactorがgormパッケージを知らずに済む — Interactorでgorm.ErrRecordNotFoundを判定すると、インフラ詳細への依存が生まれます。」

と述べています。しかし、直後の「✅ 正しい例」として示されるInteractorのコードでは：

```go
func isDuplicateKeyError(err error) bool {
    var pgErr *pgconn.PgError
    return errors.As(err, &pgErr) && pgErr.Code == "23505"
}
```

Interactorが **PostgreSQL固有の `pgconn.PgError`** に直接依存しています。これは「インフラ詳細への依存を避ける」という主張と矛盾しており、読者に誤解を与えます。この設計上のトレードオフを記事内で明示すべきです。

### 🟡 エビデンスなしに確定的に述べている箇所

#### 3. 「教科書的には〜とされています」の出典が不明

> 「教科書的には『インフラ層はインフラエラーをそのまま返し、アプリケーション層で解釈すべき』とされています。」

これは参考文献に挙げている Vaughn Vernon の本など具体的などの文献でそのように述べられているのかが示されておらず、「教科書的」という表現で既定事実のように書かれています。該当する引用元を明示するか、「一般的にそのように解釈されることが多い」程度の表現に抑えるほうが正確です。

### ✅ 正確な部分

- PostgreSQLの一意制約違反エラーコード `"23505"` は正確です
- `errors.As` を使った `HandleError` の型判定の説明は正確です
- `gorm.ErrRecordNotFound` を `model.ErrTaskNotFound` に変換する Repository パターンの説明は技術的に正確です
- セキュリティセクション（404 vs 403、スタックトレース、ログ分離）の内容は適切です

# Smart Commit with Gitmoji

Execute the following steps non-interactively:

## Branch Management

- If currently on `main` or `master` branch, create and checkout a new feature branch with a descriptive name
- If on any other branch, proceed with commit on current branch (no branch creation)
- If no changes are detected, exit without doing anything

## Change Analysis & Commit

1. **Review all changes** using `git status` and `git diff --staged` (or `git diff` if nothing staged)
2. **Stage changes** if not already staged using `git add .` or selective staging
3. **Analyze the changes** to understand:
   - Type of changes (new feature, bug fix, refactor, etc.)
   - Scope of changes (which files/components affected)
   - Impact and purpose of the changes

## Commit Message Creation

Create a commit message following this format:

```
<gitmoji> <type>: <concise description>

<optional detailed explanation if needed>
```

### Gitmoji Selection Guide

Choose the most appropriate gitmoji from <https://gitmoji.dev/> based on change type:

**Common Patterns:**

- ✨ `:sparkles:` - New features
- 🐛 `:bug:` - Bug fixes
- 🎨 `:art:` - Code structure/format improvements
- ⚡️ `:zap:` - Performance improvements
- 📝 `:memo:` - Documentation updates
- 🔧 `:wrench:` - Configuration changes
- ♻️ `:recycle:` - Code refactoring
- 🚨 `:rotating_light:` - Fix linter warnings
- ✅ `:white_check_mark:` - Add/update tests
- 🔒️ `:lock:` - Security fixes
- ⬆️ `:arrow_up:` - Upgrade dependencies
- 🚚 `:truck:` - Move/rename files
- 🗑️ `:wastebasket:` - Remove deprecated code

### Message Guidelines

- **Title**: Keep under 50 characters, use imperative mood
- **Body**: Explain "what" and "why", not "how"
- **Language**: Write in English
- **Tone**: Professional and concise

## Final Steps

- Execute the commit with the generated message
- Display a summary of what was committed
- Suggest next steps (push to remote, create PR, etc.)

## 否定的レビュー

### 🔴 論理の破綻：記事内でルールが自己矛盾している

#### 「変換すべきでないケース」と「✅正しい例」が矛盾する

記事はまず明確にこう述べています：
> 「文脈によって意味が変わるエラー変換はインフラ層で行うべきではありません」

ところが直後の「✅ インフラ層は事実をドメインの言葉で返す」で、

```go
func (r *userRepository) Save(...) error {
    if isDuplicateKeyError(result.Error) {
        return model.ErrDuplicateEmail // インフラ層でドメインエラーに変換
    }
}
```

これはまさにインフラ層でドメインエラーに変換しています。記事はこれを「userRepositoryはUserに特化しているから一意に決まる」と弁明しますが、この論理は先に示した ❌ 例の汎用Repositoryとの違いを「Repository名が特定か汎用か」という実装詳細に依存させており、**判断基準がぼやけています**。読者は「結局どこで変換すればいいのか」を迷うことになります。

---

### 🟡 実用上の重大な懸念が1行で片付けられている

#### レースコンディションの軽視

Interactor層で「事前チェック → 保存」パターンを ✅ として提示しておきながら、

> 「ただし、事前チェックと保存の間でレースコンディションを起こす恐れがあります。トランザクションやユニーク制約との併用を検討してください。」

これは本番環境で致命的になりうる問題を1文で流しています。「ユニーク制約と併用」するなら、結局DB側の一意制約違反エラーをどこかで処理する必要があり、**この ✅ パターンは問題を先送りにしているだけ**で解決していません。補足的な注意書きではなく、このトレードオフを正面から議論すべきです。

---

### 🟡 定義された変数が説明なく使われていない

#### `ErrInternal`（SYS001）の存在意義が不明

`apperror/codes.go` で以下が定義されています：

```go
ErrInternal = newCodedError("SYS001", "internal error", http.StatusInternalServerError)
```

しかし `HandleError` ではハンドルされないエラーに対してハードコードの `"SYS999"` を返しており、`ErrInternal` はコード例の中で一度も使われません。「SYS001とSYS999はどう使い分けるのか」が説明されておらず、読者は両者の役割の違いを判断できません。

---

### 🟡 参考文献の選定が時代遅れ

#### Dave Cheneyの記事（2016年）は現行のGoエラーハンドリングを反映していない

参考文献に挙げている「Don't just check errors, handle them gracefully」は2016年の記事で、`errors.Is` / `errors.As` が導入された **Go 1.13（2019年）以前** の内容です。記事の中心的なパターンである `errors.As` による型判定はこの文献では扱われていません。現行の実装を裏付ける文献として引用するには不適切で、「現在はより良いアプローチがある」といった但し書きが必要です。

---

### 🟡 セキュリティセクションが実装例を欠いている

#### 「一律404を返す」の具体的な実装が示されていない

> 「認可エラーでもリソースの存在を隠したい場合は、一律404を返す設計を検討してください。」

これは重要な設計判断ですが、コード例がありません。`CodedError`の設計に組み込むのか、`HandleError`で吸収するのか、Interactor層で判断するのか、読者は実装方法を自分で考えるしかありません。他のセクションと比べて記述が薄く、このセクションだけ抽象論で終わっています。

---

### 🟢 問題にはなっていないが設計上の盲点

#### `CodedError` を `fmt.Errorf("%w", apperror.ErrNotFound)` でラップした場合の挙動が未説明

記事は `errors.As` と `errors.Is` の使い分けを説明していますが、Interactorが `CodedError` を `fmt.Errorf` でラップして返した場合（`fmt.Errorf("failed: %w", apperror.ErrNotFound)`）、`errors.As` は依然として `*CodedError` を取得できます。この挙動を知らない読者が「ラップしたらHandleErrorが効かなくなる」と誤解して不必要なコードを書くリスクがあります。

---

### まとめ

| 深刻度 | 問題 |
|--------|------|
| 🔴 高 | 「変換すべきでないケース」のルールと直後の正例が自己矛盾している |
| 🟡 中 | レースコンディションの問題が1行で片付けられており解決していない |
| 🟡 中 | `ErrInternal`（SYS001）の使い所が未説明 |
| 🟡 中 | Dave Cheney引用はGo 1.13以前の内容で現在の実装を裏付けない |
| 🟡 中 | 「一律404」の実装方法が示されていない |

最も優先すべき修正は**「変換すべきでないケース」と「✅ userRepository.Save」の間の論理的整合性**です。現状では読者が判断基準を正確に把握できません。

# ドキュメントレビュー

引数で指定された Markdown 記事ファイルの文章品質をレビューし、問題を指摘したうえで自動修正する。

## 入力

@articles/56e161aff29ff9.md に対象ファイルのパスが渡される（例: `articles/978121945958ed.md`）。パスが渡されない場合は「対象ファイルを指定してください（例: /review-doc articles/xxx.md）」とだけ返して終了すること。

## レビュー手順

### Step 1: 記事を読み込む

対象ファイルを Read ツールで全文読み込むこと。

### Step 2: 以下の 4 観点でレビューする

#### 観点 A: 冗長・繰り返し

- 同じ主張が複数箇所で繰り返されていないか
- 削れる前置き・つなぎ表現はないか（「〜がある。それは〜である。」のような回りくどい表現）
- 1 文が長すぎないか（目安: 80 文字超は分割を検討）

#### 観点 B: 論理構成・流れ

- セクション間の論理的なつながりが自然か
- 見出し階層が適切か（h2 → h3 の飛び越しがないか）
- 結論が導入の問いに対応しているか

#### 観点 C: 読者価値密度

- 各セクションに具体例・数値・コード例など「読んで良かった」と思える情報があるか
- 抽象的な主張だけで終わっているセクションはないか
- 読者が次のアクションを取れる情報が含まれているか

#### 観点 D: 文体の統一

- 記事全体が敬語（ですます調）で統一されているか
- 箇条書き内で「である」調が混在していないか
- 見出し直下の地の文とリスト項目で文体が揺れていないか

### Step 3: 指摘を出力する

以下のフォーマットで指摘を一覧表示すること。

```markdown
## レビュー結果: <ファイル名>

### 観点 A: 冗長・繰り返し

- [ ] L<行番号>: <指摘内容>（<改善案の要約>）

### 観点 B: 論理構成・流れ

- [ ] L<行番号>: <指摘内容>（<改善案の要約>）

### 観点 C: 読者価値密度

- [ ] L<行番号>: <指摘内容>（<改善案の要約>）

### 観点 D: 文体の統一

- [ ] L<行番号>: <指摘内容>（<改善案の要約>）

### 総評

<全体の品質に関する 2〜3 行のコメント>
```

指摘がない観点は「問題なし」とだけ記載すること。

### Step 4: 自動修正を適用する

指摘のうち、意味を変えずに改善できるもの（冗長表現の削除、文の分割など）は Edit ツールで自動修正すること。修正を適用したら、修正箇所のサマリーを出力すること。
