:::message

本記事は私の実装経験と考察をベースにしています。各セクションの根拠となる一次情報源は、該当箇所に参照リンクを記載しています。

:::

DDDで設計されたGoのプロジェクトで、エラーハンドリングに悩んだことはないでしょうか。

- ドメイン層のバリデーションエラーとDBの接続エラーが同じ`error`として返される
- Handler層で`err != nil`の中身を見て、400なのか500なのかを判断するif文が増殖する
- エラーメッセージをそのままクライアントに返してしまい、内部構造が漏洩する

この記事では、DDDのレイヤー構造に沿った**エラーの分類・設計・変換パターン**を整理し、Goの`error` interfaceを活かした実装を示します。

---

## エラーの3分類

DDDのレイヤーに対応させると、エラーは大きく3つに分類できます。

| 分類 | 発生レイヤー | 例 | HTTPステータス |
| --- | --- | --- | --- |
| ドメインエラー | Domain層 | 「在庫が不足しています」「メールアドレスの形式が不正です」 | 400 / 409 / 422 |
| アプリケーションエラー | Application層 | 「指定されたユーザーが見つかりません」「権限がありません」 | 404 / 403 |
| インフラエラー | Infrastructure層 | 「DBに接続できません」「外部APIがタイムアウトしました」 | 500 / 502 / 503 |

重要なのは、**上位レイヤーが下位レイヤーのエラーの詳細を知らなくてよい**ようにすることです。Handler層は「ドメインエラーか、インフラエラーか」を区別できれば十分で、SQLのエラーコードを知る必要はありません。

## ドメインエラーの設計

ドメインエラーは**ビジネスルール違反**を表します。ドメインの言葉で記述し、インフラの詳細を含めません。

### パターン1：sentinel errorを使う

最もシンプルなパターンです。エラーの種類が少なく、付加情報が不要な場合に適しています。

```go
// domain/order/errors.go
package order

import "errors"

var (
    ErrInsufficientStock = errors.New("在庫が不足しています")
    ErrAlreadyCancelled  = errors.New("この注文はすでにキャンセルされています")
    ErrEmptyItems        = errors.New("注文には1つ以上の商品が必要です")
)
```

```go
// domain/order/order.go
func (o *Order) Cancel() error {
    if o.status == StatusCancelled {
        return ErrAlreadyCancelled
    }
    if o.status == StatusShipped {
        return errors.New("発送済みの注文はキャンセルできません")
    }
    o.status = StatusCancelled
    return nil
}
```

利用側では`errors.Is`で判定します。

```go
// application/cancel_order.go
err := order.Cancel()
if errors.Is(err, order.ErrAlreadyCancelled) {
    // 冪等に扱う（成功として返す）
    return nil
}
```

### パターン2：カスタムエラー型を使う

エラーに付加情報（どのフィールドが不正か、どの値が上限を超えたか等）を持たせたい場合に使います。

```go
// domain/shared/domain_error.go
package shared

// DomainError はビジネスルール違反を表すエラーです。
type DomainError struct {
    Code    ErrorCode
    Message string
    Field   string // バリデーションエラーの場合、対象フィールド
}

func (e *DomainError) Error() string {
    return e.Message
}

type ErrorCode string

const (
    ErrCodeValidation   ErrorCode = "VALIDATION_ERROR"
    ErrCodeConflict     ErrorCode = "CONFLICT"
    ErrCodePrecondition ErrorCode = "PRECONDITION_FAILED"
)
```

```go
// domain/user/email.go
package user

import "example/internal/domain/shared"

type Email struct {
    value string
}

func NewEmail(value string) (Email, error) {
    if !isValidEmail(value) {
        return Email{}, &shared.DomainError{
            Code:    shared.ErrCodeValidation,
            Message: "メールアドレスの形式が不正です",
            Field:   "email",
        }
    }
    return Email{value: value}, nil
}
```

利用側では`errors.As`で型アサーションします。

```go
var domainErr *shared.DomainError
if errors.As(err, &domainErr) {
    // ドメインエラーとして処理
    log.Printf("ドメインエラー: field=%s, code=%s", domainErr.Field, domainErr.Code)
}
```

### どちらを選ぶか

| 判断基準         | sentinel error    | カスタムエラー型                   |
| ---------------- | ----------------- | ---------------------------------- |
| エラーの種類     | 少ない（5個以下） | 多い、または増える可能性がある     |
| 付加情報         | 不要              | フィールド名、上限値等が必要       |
| 複数エラーの集約 | 不向き            | バリデーション結果をまとめて返せる |

実務では**併用**するのが現実的です。単純なビジネスルール違反にはsentinel error、バリデーション系にはカスタムエラー型を使います。

## アプリケーションエラーの設計

アプリケーション層では、ユースケース固有のエラーが発生します。代表的なのは「リソースが見つからない」「認可エラー」です。

```go
// application/errors.go
package application

import "fmt"

type NotFoundError struct {
    Resource string
    ID       string
}

func (e *NotFoundError) Error() string {
    return fmt.Sprintf("%sが見つかりません: %s", e.Resource, e.ID)
}

type ForbiddenError struct {
    Action string
    Reason string
}

func (e *ForbiddenError) Error() string {
    return fmt.Sprintf("%sの権限がありません: %s", e.Action, e.Reason)
}
```

```go
// application/get_user.go
func (uc *GetUserUseCase) Execute(ctx context.Context, id user.UserID) (*UserDTO, error) {
    u, err := uc.finder.FindByID(ctx, id)
    if err != nil {
        return nil, err // インフラエラーはそのまま上位へ
    }
    if u == nil {
        return nil, &NotFoundError{Resource: "ユーザー", ID: id.String()}
    }
    return toDTO(u), nil
}
```

ここで重要なのは、**infrastructure層から返されたエラーをアプリケーション層で「翻訳」するかどうか**です。

## インフラエラーの扱い方

### 原則：インフラエラーはラップして上位に伝播する

```go
// infrastructure/user/mysql_repository.go
func (r *MySQLUserRepository) FindByID(ctx context.Context, id user.UserID) (*user.User, error) {
    row := r.db.QueryRowContext(ctx, "SELECT id, name, email FROM users WHERE id = ?", id.String())
    var dest userRow
    if err := row.Scan(&dest.ID, &dest.Name, &dest.Email); err != nil {
        if errors.Is(err, sql.ErrNoRows) {
            return nil, nil // 見つからない場合はnilを返す（エラーではない）
        }
        return nil, fmt.Errorf("ユーザーの取得に失敗しました: %w", err)
    }
    return toEntity(&dest), nil
}
```

`%w`でラップすることで、上位レイヤーが必要に応じて`errors.Is`や`errors.As`で元のエラーを検査できます。

### アンチパターン：インフラエラーをドメインエラーに変換する

```go
// ❌ これはやらない
func (r *MySQLUserRepository) Save(ctx context.Context, u *user.User) error {
    _, err := r.db.ExecContext(ctx, "INSERT INTO users ...")
    if err != nil {
        if isDuplicateKeyError(err) {
            // インフラ層がドメインエラーを生成してしまっている
            return &shared.DomainError{
                Code:    shared.ErrCodeConflict,
                Message: "このメールアドレスは既に使用されています",
            }
        }
        return err
    }
    return nil
}
```

この判断はアプリケーション層の責務です。infrastructure層は「一意制約違反が起きた」という事実をラップして返し、アプリケーション層がそれを解釈します。

```go
// ✅ infrastructure層：事実を返す
func (r *MySQLUserRepository) Save(ctx context.Context, u *user.User) error {
    _, err := r.db.ExecContext(ctx, "INSERT INTO users ...")
    if err != nil {
        return fmt.Errorf("ユーザーの保存に失敗しました: %w", err)
    }
    return nil
}

// ✅ application層：事実を解釈してドメインの文脈で返す
func (uc *CreateUserUseCase) Execute(ctx context.Context, input CreateUserInput) error {
    u, err := user.New(input.Name, input.Email)
    if err != nil {
        return err // ドメインエラー
    }
    if err := uc.repo.Save(ctx, u); err != nil {
        if isDuplicateKeyError(err) {
            return &shared.DomainError{
                Code:    shared.ErrCodeConflict,
                Message: "このメールアドレスは既に使用されています",
                Field:   "email",
            }
        }
        return err // インフラエラー
    }
    return nil
}
```

:::message

ドメイン層で重複チェックを行う（`DuplicateChecker`ドメインサービス）方がDDDとしては理想的です。しかし、DB制約と二重チェックになるため、実務ではアプリケーション層でDB制約のエラーを解釈するパターンも現実的な選択肢です。

:::

## Handler層でのエラー変換

すべてのエラーは最終的にHandler層（Presentation層）でHTTPレスポンスに変換されます。ここが**エラー分類の成果を回収する場所**です。

```go
// presentation/error_handler.go
package presentation

import (
    "errors"
    "net/http"

    "example/internal/application"
    "example/internal/domain/shared"
)

type ErrorResponse struct {
    Code    string `json:"code"`
    Message string `json:"message"`
    Field   string `json:"field,omitempty"`
}

func HandleError(w http.ResponseWriter, err error) {
    var domainErr *shared.DomainError
    if errors.As(err, &domainErr) {
        status := domainErrorToStatus(domainErr.Code)
        writeJSON(w, status, ErrorResponse{
            Code:    string(domainErr.Code),
            Message: domainErr.Message,
            Field:   domainErr.Field,
        })
        return
    }

    var notFound *application.NotFoundError
    if errors.As(err, &notFound) {
        writeJSON(w, http.StatusNotFound, ErrorResponse{
            Code:    "NOT_FOUND",
            Message: notFound.Error(),
        })
        return
    }

    var forbidden *application.ForbiddenError
    if errors.As(err, &forbidden) {
        writeJSON(w, http.StatusForbidden, ErrorResponse{
            Code:    "FORBIDDEN",
            Message: forbidden.Error(),
        })
        return
    }

    // それ以外はすべてインフラエラー（内部エラー）
    // 詳細はログに記録し、クライアントには汎用メッセージを返す
    slog.Error("内部エラー", "error", err)
    writeJSON(w, http.StatusInternalServerError, ErrorResponse{
        Code:    "INTERNAL_ERROR",
        Message: "内部エラーが発生しました",
    })
}

func domainErrorToStatus(code shared.ErrorCode) int {
    switch code {
    case shared.ErrCodeValidation:
        return http.StatusBadRequest
    case shared.ErrCodeConflict:
        return http.StatusConflict
    case shared.ErrCodePrecondition:
        return http.StatusPreconditionFailed
    default:
        return http.StatusBadRequest
    }
}
```

```go
// presentation/user/handler.go
func (h *UserHandler) CreateUser(w http.ResponseWriter, r *http.Request) {
    var input application.CreateUserInput
    if err := json.NewDecoder(r.Body).Decode(&input); err != nil {
        writeJSON(w, http.StatusBadRequest, ErrorResponse{
            Code:    "INVALID_REQUEST",
            Message: "リクエストの形式が不正です",
        })
        return
    }

    if err := h.createUser.Execute(r.Context(), input); err != nil {
        HandleError(w, err)
        return
    }

    w.WriteHeader(http.StatusCreated)
}
```

Handler層のコードが**シンプルかつ安全**になります。

## エラーの流れ全体像

```text
【ドメインエラー】
Domain層で発生 → Application層をそのまま通過 → Handler層でHTTP 4xx

【アプリケーションエラー】
Application層で発生 → Handler層でHTTP 4xx

【インフラエラー】
Infrastructure層で発生 → Application層をそのまま通過 → Handler層でHTTP 5xx
（ただし一部のインフラエラーはApplication層でドメインエラーに変換される）
```

```text
Domain層          Application層          Presentation層

ErrEmptyItems ──→ そのまま通過 ────────→ 400 Bad Request

                  NotFoundError ───────→ 404 Not Found

                  DB unique違反 ───→
                  DomainError(Conflict) → 409 Conflict

sql.ErrConnDone → そのまま通過 ────────→ 500 Internal Error
                                        （ログ記録、詳細は非公開）
```

## セキュリティの観点

エラーハンドリングは**セキュリティの多層防御**の一部でもあります。

### 守るべきルール

1. **インフラエラーの詳細をクライアントに返さない**

```go
// ❌ SQLエラーがそのまま露出
writeJSON(w, 500, map[string]string{
    "error": err.Error(), // "Error 1045: Access denied for user 'root'@'localhost'"
})

// ✅ 汎用メッセージ + ログ
slog.Error("DB接続エラー", "error", err)
writeJSON(w, 500, ErrorResponse{
    Code:    "INTERNAL_ERROR",
    Message: "内部エラーが発生しました",
})
```

1. **存在確認のエラーで情報を漏らさない**

```go
// ❌ ユーザーの存在が推測できる
// POST /login → "パスワードが間違っています"（= ユーザーは存在する）

// ✅ 存在を推測させない
// POST /login → "メールアドレスまたはパスワードが間違っています"
```

1. **スタックトレースをクライアントに返さない**

本番環境では、スタックトレースはログに記録し、クライアントにはエラーコードのみを返します。

## まとめ

| レイヤー | エラーの責務 | 実装パターン |
| --- | --- | --- |
| Domain | ビジネスルール違反を表現する | sentinel error / DomainError型 |
| Application | ユースケース固有のエラー + インフラエラーの解釈 | NotFoundError / ForbiddenError |
| Infrastructure | 技術的エラーをラップして返す | `fmt.Errorf("...: %w", err)` |
| Presentation | エラー分類に基づきHTTPレスポンスに変換する | `errors.As`による型判定 |

Goの`error` interfaceはシンプルですが、**sentinel error、カスタムエラー型、`errors.Is/As`の組み合わせ**でDDDレイヤー構造に沿ったエラーハンドリングを実現できます。各レイヤーが**自分の責務に応じたエラーだけを生成し、他レイヤーのエラーは変換せず伝播する**ことが重要です。
