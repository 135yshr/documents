## はじめに

脆弱性レビューというと「危険なものを見つける」ことに注目が集まりがちです。CVE が発行されるような深刻な脆弱性を発見し、修正する。それがセキュリティレビューの花形であることは間違いありません。

しかし「安全と判断できたもの」にも重要な学びがあります。なぜ安全なのかを構造的に説明できることは、なぜ危険なのかを説明することと同じくらい価値があります。

本記事では、自作トランスパイラ [meow](https://github.com/135yshr/meow) のセキュリティレビューで指摘を受けたものの、実際には安全だった 2 つのケースを分析します。そしてそこから見える多層防御（Defense in Depth）の効果を具体的に見ていきます。

meow は Go で書かれたコード生成ツールで、独自のソースファイルを Go のソースコードに変換し、`go build` でバイナリを生成します。ユーザーの入力がコード生成を経て最終的に実行されるため、セキュリティ上の懸念が複数のポイントで発生し得ます。

## ケース 1: Unicode ホモグリフによる変数名偽装

### ケース 1 の指摘内容

「Unicode のホモグリフ（見た目が同じ別の文字）を使って変数名を偽装できるのではないか」という指摘です。

具体例を挙げます。

- `a` (U+0061, Latin Small Letter A) と `а` (U+0430, Cyrillic Small Letter A)
- `o` (U+006F, Latin Small Letter O) と `о` (U+043E, Cyrillic Small Letter O)

画面上ではまったく同じに見えますが、Unicode のコードポイントが異なるため、プログラム上は別の文字として扱われます。

```go
var price = 100       // Latin の 'p'
var рrice = 0         // Cyrillic の 'р'（見た目は同じ）
fmt.Println(price)    // どちらが出力される？
```

これは実際に他の言語で報告されている攻撃手法です。2021 年に公開された Trojan Source（CVE-2021-42574）では、Unicode の BiDi 制御文字を使った攻撃手法が示されました。ホモグリフもソースコードの視覚的偽装に利用される関連する攻撃手法です。レビュアーには正常に見えますが、実際には悪意ある動作をするコードを作成できます。

### ケース 1 が安全な理由

**第 1 層: Lexer の文字種制限**

meow の lexer は、識別子として ASCII 英数字と `_` のみを受け付けます。

```go
func isIdentChar(ch rune) bool {
    return ch == '_' || ('a' <= ch && ch <= 'z') || ('A' <= ch && ch <= 'Z') || ('0' <= ch && ch <= '9')
}
```

Cyrillic やその他の Unicode 文字は lexer の段階で拒否されます。`рrice`（Cyrillic の `р` で始まる）は有効な識別子として認識されず、字句解析エラーになります。ホモグリフ攻撃はそもそも成立しません。

**第 2 層: Go コンパイラによる構文検証**

仮に lexer をバイパスして不正なコードが生成に混入したとしても、最終的なビルドを行う Go コンパイラが構文の妥当性を検証するため、構文的に不正なコードはコンパイルエラーになります。

ただし、Go は Unicode カテゴリ Lu・Ll・Lt・Lm・Lo の文字を識別子として受け入れます。有効な Unicode 文字で構成されたホモグリフは構文的には正しいため、Go コンパイラでは検出できません。つまり、第 2 層はホモグリフ攻撃には無効であり、ケース 1 の防御は第 1 層（lexer の ASCII 制限）のみが担っています。第 2 層はあくまで構文破壊に対するセーフティネットです。

### ケース1の防御構造

```text
[ソースコード] → [Lexer: ASCII のみ許可] → [Parser] → [Codegen] → [Go コンパイラ: 最終検証]
                  ^^^^^^^^^^^^^^^^^^^^^^^^                          ^^^^^^^^^^^^^^^^^^^^^^^^
                  第1層: 入口で拒否                                   第2層: 出口で検証
```

第 1 層で攻撃の入口を塞ぎ、第 2 層で万が一のすり抜けを検出します。

## ケース 2: 文字列リテラルの codegen 注入

### ケース 2 の指摘内容

「ユーザーの文字列リテラルを、コード生成時にそのまま Go ソースへ埋め込むと、コード注入ができるのではないか」という指摘です。

攻撃シナリオを考えてみます。

```go
msg := `hello"); os.Exit(1); fmt.Println("`
```

もしエスケープなしで文字列をそのまま埋め込むと、生成される Go コードは次のようになります。

```go
msg := runtime.NewString("hello"); os.Exit(1); fmt.Println("")
```

文字列リテラルの中身が Go のコードとして解釈され、任意のコードが実行されてしまいます。これはコード生成ツール全般に共通するリスクであり、SQL インジェクションと同じ構造の脆弱性です。

### ケース 2 が安全な理由

**第 1 層: `%q` フォーマットによるエスケープ**

コード生成時に Go の `fmt.Sprintf("%q", str)` を使用しています。

```go
// codegen の実装
func (g *Generator) genStringLiteral(s string) string {
    return fmt.Sprintf("runtime.NewString(%q)", s)
}
```

`%q` は Go 構文に安全な引用符付き文字列を生成します。具体的には、ダブルクォート、バックスラッシュ、改行、その他の特殊文字をすべてエスケープした上で、前後をダブルクォートで囲みます。

```go
fmt.Sprintf("%q", `hello"); os.Exit(1); fmt.Println("`)
// => "hello\"); os.Exit(1); fmt.Println(\""
```

攻撃者が注入しようとしたコードは、すべて文字列リテラルの中に閉じ込められます。バックスラッシュ、引用符、改行などすべてがエスケープされるため、文字列の外へ脱出できません。

**第 2 層: Go コンパイラのパースエラー**

`%q` のエスケープに不備があり不正な Go コードを生成してしまっても、`go build` のパースエラーで拒否されます。不正なコードの実行には至りません。

### ケース2の防御構造

```text
[ユーザー文字列] → [%q でエスケープ] → [Go ソース生成] → [Go コンパイラ: 構文検証]
                   ^^^^^^^^^^^^^^^^^                      ^^^^^^^^^^^^^^^^^^^^^^^^
                   第1層: エスケープ処理                      第2層: コンパイラ検証
```

第 1 層がエスケープ処理で注入を無効化し、第 2 層がコンパイラレベルで不正なコードの実行を阻止します。

## 多層防御（Defense in Depth）から学ぶこと

### 1 層だけに依存しない

2 つのケースとも、lexer の ASCII 制限や `%q` のエスケープにより第 1 層だけで攻撃を防いでいます。しかし、第 2 層（Go コンパイラ）が存在することで、以下の状況に備えています。

- 第 1 層にバグが見つかった場合
- 将来の変更で第 1 層の制約が緩和された場合
- 想定外の入力パスが発見された場合

たとえば将来、ユーザーの要望で Unicode 識別子をサポートすることになったとします。第 1 層の lexer 制限は緩和されますが、第 2 層の Go コンパイラの検証は引き続き機能します。もちろん Go は Unicode 識別子を許容するため、その時点で別の防御策の検討が必要です。しかし少なくとも「第 1 層の変更で即座に脆弱となる」事態は避けられます。

### 各層の役割を理解する

| 層                                 | 役割                 | 特性                               |
| ---------------------------------- | -------------------- | ---------------------------------- |
| 第 1 層（アプリケーション）        | 入力制限・エスケープ | 高速、特化型、開発者がコントロール |
| 第 2 層（コンパイラ / 外部ツール） | 最終検証             | 汎用的、高信頼、変更困難           |

第 1 層は「攻撃を防ぐ」、第 2 層は「万が一すり抜けても被害を防ぐ」という異なる目的を持ちます。

第 1 層はプロジェクト固有のロジックであり、開発者が自由に変更できます。これは柔軟性がある反面、バグが混入するリスクもあります。一方、第 2 層の Go コンパイラは何百万人もの開発者に使われ、長年にわたってテストされてきた外部ツールです。信頼性は高いですが、プロジェクト固有の要件に合わせた細かい制御はできません。

この「柔軟で脆い層」と「固定的で堅牢な層」の組み合わせこそ、多層防御の強みです。

### 多層防御が効いているか確認する方法

コードレビューやセキュリティレビューで以下の観点を確認すると、防御の健全性を評価できます。

1. **各防御層を特定する**: 入力から実行まで、どこにどんなチェックがあるかを確認します。データの流れを追い、各ポイントでの検証内容を列挙します
2. **各層が独立しているか**: 第 1 層が失敗しても第 2 層が機能するかを確認します。共通のロジックや設定に依存していると、同時に破られるリスクがあります
3. **各層の信頼性**: 外部ツール（コンパイラ等）の検証は自前の検証より一般に信頼性が高いです。自前の検証ロジックにはテストを書き、外部ツールの検証はその仕様を理解しておきます

## 「安全」の判断を記録する重要性

脆弱性が見つかった場合、修正内容は PR やコミットに記録されます。しかし「修正不要」と判断したケースは、意識的に記録しなければ消えてしまいます。

判断根拠を以下のような形式で記録しておくことを推奨します。

```markdown
### Unicode Homoglyph Attack

- **Status**: Not vulnerable
- **Reason**: Lexer restricts identifiers to ASCII only. Go compiler provides secondary validation.
- **Review trigger**: If Unicode identifier support is added in the future, this assessment must be revisited.
```

ここで重要なのは `Review trigger`（再評価のトリガー）です。「どのような変更が行われたときに、この安全性の前提が崩れるか」を明記しておくことで、将来の変更時に再評価が必要であることを開発チームに伝えられます。

セキュリティレビューの結果を Issue やドキュメントに残しておけば、半年後に新メンバーが関連する変更をする際、「この部分はセキュリティレビュー済みだが、この変更なら再評価が必要」と判断できます。

## まとめ

本記事では、meow のセキュリティレビューで「安全」と判断した 2 つのケースを通じて、多層防御の実際の効果を分析しました。

- **「なぜ安全か」を説明できること**は「なぜ危険か」の説明と同じくらい重要です。安全性の根拠を構造的に説明できなければ、それは「安全」ではなく「未検証」です
- **多層防御が効いているポイントを把握しておく**と、1 層が破られた場合のリスク評価ができます。第 1 層だけで安全でも、第 2 層の存在が将来の変更への耐性を生みます
- **セキュリティレビューの記録として「安全と判断した根拠」も残す**ことで、将来の変更時の再評価トリガーとして機能します
- **防御層の追加・変更時には、既存の安全性評価の前提が崩れていないか確認する**習慣をチームに定着させることが、多層防御を維持する鍵です

セキュリティレビューは「脆弱性を見つける行為」だけではありません。「なぜ安全なのか」を明らかにし、その安全性がどのような条件で保たれているかを記録する行為でもあります。
