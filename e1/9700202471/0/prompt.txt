## 修正① `NewEmail` のバリデーションバグ（L119–128）

**問題:** `strings.SplitN(v, "@", 2)` は最初の `@` でしか分割しないため
`user@@domain.com` → `parts[1] = "@domain.com"` となり通過してしまう。

**現状コード（L123–128）:**
```go
parts := strings.SplitN(v, "@", 2)
if len(parts) != 2 || parts[0] == "" || parts[1] == "" || !strings.Contains(parts[1], ".") {
    return Email{}, fmt.Errorf("メールアドレスの形式が不正です: %s", v)
}
return Email{value: strings.ToLower(v)}, nil
```

**修正後:**
```go
if strings.Count(v, "@") != 1 {
    return Email{}, fmt.Errorf("メールアドレスの形式が不正です: %s", v)
}
parts := strings.SplitN(v, "@", 2)
if parts[0] == "" || parts[1] == "" || !strings.Contains(parts[1], ".") {
    return Email{}, fmt.Errorf("メールアドレスの形式が不正です: %s", v)
}
normalized := parts[0] + "@" + strings.ToLower(parts[1])
return Email{value: normalized}, nil
```

※ `Domain()` メソッドは `e.value` から再分割しているため、変更不要（ドメイン部はすでに小文字化済み）。

## 修正③ `Role` の文字列ハードコーディング（L293–305）

**問題:** `CanEdit()` と `IsAdmin()` で `"admin"` 等のリテラルを直書き。
`OrderStatus` が `iota` 定数で一貫しているのに `Role` だけ文字列直書きで矛盾している。

**現状コード（L292–305付近）:**
```go
func NewRole(v string) (Role, error) {
    allowed := map[string]bool{"admin": true, "editor": true, "viewer": true}
    ...
}
func (r Role) CanEdit() bool {
    return r.value == "admin" || r.value == "editor"
}
func (r Role) IsAdmin() bool {
    return r.value == "admin"
}
```

**修正後:** `NewRole` の前に定数ブロックを追加し、メソッドと `NewRole` で参照する。

```go
const (
    roleAdmin  = "admin"
    roleEditor = "editor"
    roleViewer = "viewer"
)

func NewRole(v string) (Role, error) {
    allowed := map[string]bool{roleAdmin: true, roleEditor: true, roleViewer: true}
    ...
}
func (r Role) CanEdit() bool {
    return r.value == roleAdmin || r.value == roleEditor
}
func (r Role) IsAdmin() bool {
    return r.value == roleAdmin
}
```

## 修正④ `CanTransitionTo` のマップ毎回生成（L357–370）

**問題:** メソッド呼び出しのたびにマップをヒープ確保する。
ステートマシンのチェックは頻繁に呼ばれる可能性があり、GC 負荷になる。

**修正後:** パッケージレベル変数に移動する。

```go
var orderStatusTransitions = map[OrderStatus][]OrderStatus{
    OrderStatusDraft:     {OrderStatusConfirmed, OrderStatusCancelled},
    OrderStatusConfirmed: {OrderStatusShipped, OrderStatusCancelled},
    OrderStatusShipped:   {OrderStatusDelivered},
    OrderStatusDelivered: {},
    OrderStatusCancelled: {},
}

func (s OrderStatus) CanTransitionTo(next OrderStatus) bool {
    for _, allowed := range orderStatusTransitions[s] {
        if allowed == next {
            return true
        }
    }
    return false
}
```

## 修正⑤ 等値比較の解説追加（L245「struct による値オブジェクト」セクション直後）

**問題:** 記事冒頭の表に「値が等しければ同一とみなす」と記載しているが、

**対応:** 「struct による値オブジェクト」節の直後に以下の説明ブロックを追加する。

```markdown
比較可能なフィールドのみからなる struct は `==` 演算子で比較できます。

\```go
e1, _ := NewEmail("user@example.com")
e2, _ := NewEmail("user@example.com")
fmt.Println(e1 == e2) // true
\```

非公開フィールドでも同一パッケージ内では `==` が使えます。
`map` や `slice` をフィールドに持つ場合は `==` が使えないため、その場合は `Equals` メソッドを実装します。
```

# Smart Commit with Gitmoji

Execute the following steps non-interactively:

## Branch Management

- If currently on `main` or `master` branch, create and checkout a new feature branch with a descriptive name
- If on any other branch, proceed with commit on current branch (no branch creation)
- If no changes are detected, exit without doing anything

## Change Analysis & Commit

1. **Review all changes** using `git status` and `git diff --staged` (or `git diff` if nothing staged)
2. **Stage changes** if not already staged using `git add .` or selective staging
3. **Analyze the changes** to understand:
   - Type of changes (new feature, bug fix, refactor, etc.)
   - Scope of changes (which files/components affected)
   - Impact and purpose of the changes

## Commit Message Creation

Create a commit message following this format:

```
<gitmoji> <type>: <concise description>

<optional detailed explanation if needed>
```

### Gitmoji Selection Guide

Choose the most appropriate gitmoji from <https://gitmoji.dev/> based on change type:

**Common Patterns:**

- ✨ `:sparkles:` - New features
- 🐛 `:bug:` - Bug fixes
- 🎨 `:art:` - Code structure/format improvements
- ⚡️ `:zap:` - Performance improvements
- 📝 `:memo:` - Documentation updates
- 🔧 `:wrench:` - Configuration changes
- ♻️ `:recycle:` - Code refactoring
- 🚨 `:rotating_light:` - Fix linter warnings
- ✅ `:white_check_mark:` - Add/update tests
- 🔒️ `:lock:` - Security fixes
- ⬆️ `:arrow_up:` - Upgrade dependencies
- 🚚 `:truck:` - Move/rename files
- 🗑️ `:wastebasket:` - Remove deprecated code

### Message Guidelines

- **Title**: Keep under 50 characters, use imperative mood
- **Body**: Explain "what" and "why", not "how"
- **Language**: Write in English
- **Tone**: Professional and concise

## Final Steps

- Execute the commit with the generated message
- Display a summary of what was committed
- Suggest next steps (push to remote, create PR, etc.)
