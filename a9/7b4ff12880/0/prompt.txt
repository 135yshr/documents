# ドキュメントレビュー

引数で指定された Markdown 記事ファイルの文章品質をレビューし、問題を指摘したうえで自動修正する。

## 入力

@articles/860e12b4a2698a.md に対象ファイルのパスが渡される（例: `articles/978121945958ed.md`）。パスが渡されない場合は「対象ファイルを指定してください（例: /review-doc articles/xxx.md）」とだけ返して終了すること。

## レビュー手順

### Step 1: 記事を読み込む

対象ファイルを Read ツールで全文読み込むこと。

### Step 2: 以下の 4 観点でレビューする

#### 観点 A: 冗長・繰り返し

- 同じ主張が複数箇所で繰り返されていないか
- 削れる前置き・つなぎ表現はないか（「〜がある。それは〜である。」のような回りくどい表現）
- 1 文が長すぎないか（目安: 80 文字超は分割を検討）

#### 観点 B: 論理構成・流れ

- セクション間の論理的なつながりが自然か
- 見出し階層が適切か（h2 → h3 の飛び越しがないか）
- 結論が導入の問いに対応しているか

#### 観点 C: 読者価値密度

- 各セクションに具体例・数値・コード例など「読んで良かった」と思える情報があるか
- 抽象的な主張だけで終わっているセクションはないか
- 読者が次のアクションを取れる情報が含まれているか

#### 観点 D: 文体の統一

- 記事全体が敬語（ですます調）で統一されているか
- 箇条書き内で「である」調が混在していないか
- 見出し直下の地の文とリスト項目で文体が揺れていないか

### Step 3: 指摘を出力する

以下のフォーマットで指摘を一覧表示すること。

```markdown
## レビュー結果: <ファイル名>

### 観点 A: 冗長・繰り返し

- [ ] L<行番号>: <指摘内容>（<改善案の要約>）

### 観点 B: 論理構成・流れ

- [ ] L<行番号>: <指摘内容>（<改善案の要約>）

### 観点 C: 読者価値密度

- [ ] L<行番号>: <指摘内容>（<改善案の要約>）

### 観点 D: 文体の統一

- [ ] L<行番号>: <指摘内容>（<改善案の要約>）

### 総評

<全体の品質に関する 2〜3 行のコメント>
```

指摘がない観点は「問題なし」とだけ記載すること。

### Step 4: 自動修正を適用する

指摘のうち、意味を変えずに改善できるもの（冗長表現の削除、文の分割など）は Edit ツールで自動修正すること。修正を適用したら、修正箇所のサマリーを出力すること。

内容の追加や構成の大幅な変更が必要な指摘は、修正案を提示するにとどめ、自動修正しないこと。

### Step 5: lint を実行する

修正後に以下を順に実行し、エラーがあれば修正すること。

```bash
npx prettier --write <対象ファイル>
npx markdownlint-cli2 <対象ファイル>
npx textlint <対象ファイル>
```

## 制約

- フロントマター（title, emoji, topics, published など）は変更しないこと
- 技術用語・固有名詞はそのまま残すこと
- 修正は最小限にとどめ、筆者の文体を尊重すること
- published の値は変更しないこと

# Smart Commit with Gitmoji

Execute the following steps non-interactively:

## Branch Management

- If currently on `main` or `master` branch, create and checkout a new feature branch with a descriptive name
- If on any other branch, proceed with commit on current branch (no branch creation)
- If no changes are detected, exit without doing anything

## Change Analysis & Commit

1. **Review all changes** using `git status` and `git diff --staged` (or `git diff` if nothing staged)
2. **Stage changes** if not already staged using `git add .` or selective staging
3. **Analyze the changes** to understand:
   - Type of changes (new feature, bug fix, refactor, etc.)
   - Scope of changes (which files/components affected)
   - Impact and purpose of the changes

## Commit Message Creation

Create a commit message following this format:

```
<gitmoji> <type>: <concise description>

<optional detailed explanation if needed>
```

### Gitmoji Selection Guide

Choose the most appropriate gitmoji from <https://gitmoji.dev/> based on change type:

**Common Patterns:**

- ✨ `:sparkles:` - New features
- 🐛 `:bug:` - Bug fixes
- 🎨 `:art:` - Code structure/format improvements
- ⚡️ `:zap:` - Performance improvements
- 📝 `:memo:` - Documentation updates
- 🔧 `:wrench:` - Configuration changes
- ♻️ `:recycle:` - Code refactoring
- 🚨 `:rotating_light:` - Fix linter warnings
- ✅ `:white_check_mark:` - Add/update tests
- 🔒️ `:lock:` - Security fixes
- ⬆️ `:arrow_up:` - Upgrade dependencies
- 🚚 `:truck:` - Move/rename files
- 🗑️ `:wastebasket:` - Remove deprecated code

### Message Guidelines

- **Title**: Keep under 50 characters, use imperative mood
- **Body**: Explain "what" and "why", not "how"
- **Language**: Write in English
- **Tone**: Professional and concise

## Final Steps

- Execute the commit with the generated message
- Display a summary of what was committed
- Suggest next steps (push to remote, create PR, etc.)

## 悲観的レビュー：GoのRepositoryパターン再考

### 🔴 致命的な自己矛盾

**sqlcセクションの内部矛盾**

記事は次のように述べています：

> 「sqlcが生成する Querier interface をそのまま使えば、Repository interfaceを**自前で定義する必要がありません**」

しかし、その直後のコード例では：

```go
// usecase/task_interactor.go
// sqlc の Querier を利用側で必要なメソッドだけに絞る
type taskQuerier interface {  // ← 自前でinterfaceを定義している
    GetTask(ctx context.Context, id string) (db.Task, error)
    ListTasksByStatus(ctx context.Context, status string) ([]db.Task, error)
}
```

「自前で定義する必要がない」と言いながら、直後に自前のinterfaceを定義しています。この矛盾は読者を混乱させます。正確には「sqlcの生成した`Querier`をそのまま使う方法」と「利用側で絞り込んだinterfaceを定義する方法」の**二択**が存在するのに、一方を否定しながらもう一方を実演しています。

---

### 🔴 アーキテクチャ上の問題を埋め込んだコード例

「段階的リファクタリング」のStep 1：

```go
type TaskInteractor struct {
    repo *postgres.TaskRepository // 具体型を直接参照
}
```

これはusecase層がinfrastructure層（`postgres`パッケージ）の具体型を直接参照しており、クリーンアーキテクチャの依存関係ルール違反です。「Goらしいアプローチ」として推奨しているにもかかわらず、シリーズ記事で積み上げてきたアーキテクチャ原則に反するコードが堂々と例示されています。

---

### 🟠 重要な技術的不足・誤謬

**1. entのテスト手段が不完全**

> 「entのクライアントは構造体なので、テスト時の差し替えにはinterfaceの定義が必要です」

entには公式の`enttest`パッケージがあり、SQLiteインメモリDBを使ったテスト用クライアントを容易に生成できます。interfaceを定義しなくても高速なテストが書けるという重要な選択肢が完全に無視されています。

**2. GORMのテスト代替手段の欠如**

GORMについて「`*gorm.DB`の直接依存はテスト困難」とし、レベル1を推奨していますが、GORMはSQLite（インメモリ）との組み合わせで単体テストに近い速度でテストができます。`sqlmock`も選択肢です。「テスト困難」という断言は不公平です。

**3. `database/sql`の推奨根拠が薄い**

> 「低レベルAPIのため抽象化の恩恵が大きい」

これだけでは理由として不十分です。`sqlmock`（`github.com/DATA-DOG/go-sqlmock`）を使えばinterfaceなしでも単体テストが書けます。この重要なエコシステムが完全に無視されています。

---

### 🟠 「3人以上」基準に根拠なし

実務での判断基準の表：

| 質問 | Yes | No |
|------|-----|-----|
| チームに3人以上の開発者がいるか | レベル1 or 2 | 自由 |

なぜ「3人」なのか？2人と3人で何が変わるのか？何の根拠も示されていません。読者が自プロジェクトに適用しようとしたときに、この基準は機能しません。

---

### 🟡 軽微だが気になる問題

**1. 向いているケースの文体の不統一**

レベル1の「向いているケース」が断言口調になっています：

> 「DBの種類を将来変更する可能性があります（PostgreSQL → DynamoDB 等）」
> 「複数のストレージを使い分けます（RDB + Redis + S3）」

これは「〜がある場合」「〜する場合」と条件を示したいはずで、「あります」「使い分けます」という断言は誤読を招きます。レベル2・3の箇条書きは「〜されています」という別の文体で書かれており、一貫性がありません。

**2. 判断フローチャートの実体が不明**

見出しに「判断フローチャート」とあるが、テキスト抽出では確認できません。画像やSVGで実装されているなら適切ですが、alt textや説明テキストが存在しない場合、スクリーンリーダーユーザーや画像非表示環境では機能しません。

**3. タイトルの問い「どこまで必要か」に答えていない**

タイトルは「DB依存を剥がすのはどこまで必要か」という明快な問いかけですが、結論は「プロジェクトの状況に応じる」という判断委ねです。読者が期待する「これが答えだ」という示唆が弱く、タイトル負けしています。

**4. レベル1コード例の可視性の問題**

```go
func NewTaskRepository(db *sql.DB) *taskRepository {
```

`taskRepository`は小文字（非公開型）を返しています。外部パッケージからinterfaceを実装する型として使えるのか、interfaceを返すべきなのかが示されていません。DDD連載記事としての設計意図が見えません。

---

### 📊 総評

| カテゴリ | 評価 |
|----------|------|
| 内部一貫性 | ❌ sqlcセクションで自己矛盾 |
| 技術的正確性 | △ enttest/sqlmock等の重要な選択肢を無視 |
| アーキテクチャ整合性 | ❌ 推奨コードがシリーズの原則に違反 |
| 判断基準の根拠 | △ チーム人数の基準が恣意的 |
| タイトルと内容の整合性 | △ 問いに対する答えが弱い |

**最優先で修正すべき点**は以下の2つです：

1. sqlcセクションの「自前でinterfaceを定義する必要がない」という記述と、直後のコード例の矛盾を解消する
2. Step 1のリファクタリング例から、usecase層がinfrastructure層の具体型を参照するコードを除去するか、「クリーンアーキテクチャ適用前の例外的なステップ」であることを明記する
