## 構造・論理面の問題

**1. タイトルと内容の乖離：「実装してみた」のに動かない**

タイトルに「実装してみた」と書いていますが、この記事のコードは**実際に動作確認できる状態になっていません**。`deserializeEvent` 関数は呼び出されているのに定義がどこにもありません。`json` パッケージの import が `order.go` に書かれていない箇所で `json.Unmarshal` を使っています（`UnmarshalOrder` 関数）。「実装してみた」と言うなら、読者が `go build` して通るコードを提示すべきです。通らないコードを「実装してみた」と称するのは看板に偽りありです。

**2. `ReplayOrder` と `ReplayFrom` の重複**

267行目に `ReplayFrom` メソッドがあり、447行目に `ReplayOrder` 関数があります。やっていることはほぼ同じ（イベントを順に `apply` する）なのに、2つ存在する理由が説明されていません。読者は「どっちを使えばいいの？」と混乱します。しかも `LoadOrder` ユースケースでは両方を使い分けていますが、`ReplayOrder` の戻り値は `version=0` から始まるのに `ReplayFrom` は既存の `version` に加算していくので、スナップショットなしのケースで `ReplayOrder` を使うと**スナップショット判定の `order.Version()-fromVersion` が常に `order.Version()` になる**という暗黙の前提に依存しています。これは説明なしでは分かりません。

**3. 楽観的ロックの実装に競合条件がある**

`Save` メソッドで `SELECT COALESCE(MAX(version), 0)` してから `INSERT` していますが、このSELECTとINSERTの間に別のトランザクションがINSERTを完了する可能性があります。PostgreSQL のデフォルト分離レベルは Read Committed なので、`SELECT` の後に別トランザクションがコミットすれば同じバージョンが挿入されます。もちろん `UNIQUE (aggregate_id, version)` 制約で最終的に弾かれますが、それならアプリ側の `SELECT` + 比較は冗長であり、「楽観的ロック」の説明として不正確です。本来は `SELECT ... FOR UPDATE` か、分離レベルを Serializable にするか、あるいは UNIQUE 制約だけに頼る（その場合はアプリ側のSELECTは不要）かのいずれかを明記すべきです。記事の説明は「ベルトとサスペンダーの両方をしている」状態なのに、その自覚がありません。

**4. スナップショットの `Load` がエラーと「存在しない」を区別できない**

`SnapshotStore.Load` の戻り値が `(*Snapshot, error)` ですが、`LoadOrder` では `snap != nil` で存在チェックしています。しかしインターフェース定義上、「スナップショットが存在しない場合」に `(nil, nil)` を返すのか `(nil, ErrNotFound)` を返すのかが不明です。`err != nil` で即 return しているので、もし実装が「存在しない = エラー」として返す場合、スナップショットのない集約は永遠にロードできません。インターフェースの契約が曖昧です。

## コード品質の問題

**5. `time.Now()` がテスト不可能**

`NewOrder`、`Confirm`、`Ship` の中で `time.Now()` を直接呼んでいます。イベントソーシングにおいてタイムスタンプはイベントの一部であり、テストで再現可能であるべきです。DDD やクリーンアーキテクチャの連載を謳っていながら、テスタビリティへの配慮がゼロなのは矛盾しています。`Confirm` に至っては `time.Now()` を2回呼んでおり（`Base.Timestamp` と `ConfirmedAt`）、ナノ秒レベルで異なる値が入る可能性があります。

**6. `OrderCancelled` がイベント定義にあるのに、コマンドメソッドがない**

**7. `event.Base` を値レシーバで埋め込んでいるが、ポインタレシーバとの一貫性がない**

`Event` インターフェースのメソッドはすべて値レシーバで定義されていますが、`Order` 集約のメソッドはポインタレシーバです。これ自体は Go として正しいですが、`raise` に渡すときにイベントが値でコピーされることの意味（イミュータビリティの保証）について一切触れていません。イベントソーシングではイベントの不変性が重要な性質なのに、Go の値セマンティクスがどう関係するかという、Go 固有の設計判断の解説が欠落しています。

**8. `toEventItems` / `fromEventItems` が公開されていない理由が不明**

これらの変換関数は unexported ですが、テストで直接検証したいケースがあるはずです。また、`OrderItem`（model パッケージ）と `OrderItemDetail`（event パッケージ）が似た構造を持つのに別の型として存在する理由が説明されていません。DDD の文脈でこの分離がなぜ必要なのかを書かないと、読者には冗長に見えます。

## 嘘・紛らわしい記述

### 1. 「Go の値セマンティクスがこれを自然に保証します」は嘘（332行目）

これは**事実として誤り**です。`OrderCreated` は `Items []OrderItemDetail` というスライスを持っています。Go ではスライスは参照型であり、構造体を値コピーしてもスライスのヘッダ（ポインタ・長さ・容量）がコピーされるだけで、**背後の配列は共有されます**。

```go
created := event.OrderCreated{Items: items}
o.raise(created)
// この後 items[0].Price = 9999 とすれば、イベント内のデータも変わる
```

「値セマンティクスがイベントの不変性を自然に保証する」と書くなら、スライスやマップを含まない構造体でのみ成り立つ話です。スライスを含む構造体でこの主張をするのは、読者に誤った安心感を与えます。正確に書くなら「値型のフィールドについてはコピーで保護されるが、スライスやマップは参照が共有されるため、必要に応じてディープコピーを行うか、生成元での変更を避ける規約が必要」です。

### 2. 「実際に動くコードを共有します」は誇大表現（34行目）

前回の修正で `deserializeEvent` が追加されましたが、依然として**コンパイルが通らない状態**です。

516行目で `import "encoding/json"` が書かれていますが、これは同じ `order.go` ファイルの2つ目のコードブロックとして示されています。Go では1ファイルに `import` 宣言は1つだけです。152-187行目のコードブロック（同じ `order.go`）の `import` に `encoding/json` が含まれていないので、読者がこれらを結合すると構文エラーになります。「実際に動くコード」と言うなら、1ファイルの内容は1ブロックで完結させるか、明示的に「以下を import に追加してください」と書くべきです。

### 3. Evans の "Part IV: Strategic Design" は参照箇所として不正確（788行目）

記事中で使われている概念は「集約」「ドメインイベント」です。Evans の原著で集約（Aggregate）が扱われるのは **Part II "The Building Blocks of a Model-Driven Design"、Chapter 6** です。Part IV "Strategic Design" は境界づけられたコンテキストやコンテキストマップなどの戦略的設計パターンを扱う部分で、集約の詳細な定義はそこにはありません。さらに言えば、**ドメインイベントは Evans の原著（2003年）ではほとんど扱われていません**。ドメインイベントを体系的に論じたのは後の文献（たとえば Vernon の "Implementing Domain-Driven Design"）です。Evans を引用するなら Part II が適切であり、ドメインイベントの出典としては Evans だけでは不十分です。

### 4. CQRS が「必要になる」という書き方は紛らわしい（659行目）

「なぜ CQRS が必要になるのか」という見出しは、イベントソーシングには CQRS が必須であるかのように読めます。しかし実際には、イベントソーシングを CQRS なしで使うことは可能です（クエリ時にリプレイするだけ）。逆に CQRS はイベントソーシングなしでも使えます。記事が述べているのは「一覧表示のようなユースケースでは非効率」という実用上の問題であり、「必要になる」のではなく「効率のために組み合わせることが多い」が正確です。

### 5. 結果整合性（Eventual Consistency）への言及がゼロ

CQRS でコマンド側とクエリ側を分離した場合、読み取りモデルは**結果整合性**になります。つまり、コマンドで注文を作成した直後にクエリ側で検索しても、プロジェクションの更新が完了するまでは結果に反映されません。これはこのアーキテクチャを採用する際の**最も重要なトレードオフ**の一つですが、記事では一切触れていません。CQRS の利点だけを語って、最大の注意点を省略するのはフェアではありません。

## コードのバグ・設計上の問題

### 6. プロジェクションが `OrderCancelled` を無視している（701-724行目）

`Handle` メソッドは `OrderCreated`、`OrderConfirmed`、`OrderShipped` を処理しますが、`OrderCancelled` に対応するケースがありません。`Cancel` コマンドメソッドを前回の指摘で追加したのに、プロジェクション側の更新を忘れています。結果として、キャンセルされた注文は読み取りモデル上で永遠に「draft」や「confirmed」のままになります。

### 7. `LoadOrder` でスナップショット保存失敗が致命エラーになる（638-646行目）

スナップショットの保存に失敗すると `LoadOrder` 全体がエラーを返します。しかし `LoadOrder` は読み取り操作であり、スナップショットは性能最適化の仕組みに過ぎません。スナップショット保存の失敗で注文の読み込み自体が失敗するのは過剰です。ログ出力にとどめて `order` を正常に返すのが適切です。少なくとも、この設計判断の理由を説明すべきです。

### 8. 存在しない集約の `LoadOrder` がゼロ値の `Order` を返す

`snap == nil` かつ `events` が空の場合（つまり集約が存在しない場合）、`ReplayOrder([]event.Event{})` は `&Order{}` を返します。これは `id` が空文字、`status` がゼロ値（空の `OrderStatus`）の不正な集約です。`ErrNotFound` のようなエラーを返すべきですが、そのハンドリングがありません。

### 9. `Base.Type` フィールドの手動設定がエラーの温床

すべてのコマンドメソッドで `Type: "OrderCreated"` のように文字列を手書きしています。typo で `"OrderCreate"` と書いた場合、`deserializeEvent` の `switch` に一致せず、DB から読み出した際に「未知のイベントタイプ」エラーになります。しかもこれはコンパイル時に検出できません。定数を定義するか、イベント構造体ごとにコンストラクタを用意して `Type` を自動設定する設計が望ましいです。記事がこのリスクに触れていないのは不親切です。

### 10. `idx_events_aggregate_id` インデックスは冗長

497行目の `UNIQUE (aggregate_id, version)` 制約により、PostgreSQL は自動的に `(aggregate_id, version)` の複合インデックスを作成します。このインデックスは `aggregate_id` 単独での検索にも使えます（複合インデックスの左端の列はプレフィックスとして機能）。したがって500行目の `CREATE INDEX idx_events_aggregate_id ON events (aggregate_id)` は重複であり、書き込み性能を無駄に劣化させます。

❌ 1. 「apply は純粋関数」← ここは危険

該当箇所：

apply は副作用を持たない純粋な状態遷移です  ￼

これは厳密には誤り or 誤解を招く表現です。

なぜ危ないか
    •    o.version++ をしている時点で副作用あり
    •    o.status = ... もミュータブルな状態変更
    •    関数型の「純粋関数」とは全く違う

👉 このままだと
「apply = pure function」
と誤解される

修正案（おすすめ）

- apply は副作用を持たない純粋な状態遷移です
+ apply は外部副作用（DB・API呼び出し等）を持たない、決定的な状態遷移です

または

apply は「イベント → 状態」への決定的な変換であり、
外部依存を持たないことが重要です

👉 ポイント
    •    「純粋関数」ではなく「決定的」
    •    DDD界隈でもこっちの表現が安全

⸻

❌ 2. 「apply は常に成功する」← 強すぎる

該当：

apply は常に成功する必要があります  ￼

なぜ危険か

これは思想としては正しいが、実装としては言い過ぎ

現実には：
    •    イベントが壊れている
    •    バージョン不整合
    •    deserializeミス

👉 こういう場合は普通に失敗する

修正案

- apply は常に成功する必要があります
+ apply は「正しいイベントが入力される限り」失敗しない設計にします

または

apply は「検証済みのイベント」を前提とするため、
通常のドメインロジックとしては失敗しない設計にします

👉 これで現実との整合が取れる
